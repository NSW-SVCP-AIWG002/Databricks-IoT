# Databricks IoTシステム 要件定義ドキュメント

## 概要
このディレクトリは、Databricks IoTシステムの包括的な要件定義を含んでいます。各要件は目的別に分類され、体系的に管理されています。IoTデバイスからのデータ収集、リアルタイム処理、分析、可視化に関する機能要件、非機能要件、技術要件を網羅しています。

## ドキュメント構成

### 📝 機能要件定義書
**ファイル**: `functional-requirements.md`
**目的**: システムが「何をするか」を定義

**含まれる内容**:
- 機能一覧（6機能）
  - FR-001: ログイン認証機能（認証共通モジュール: Azure/AWS/オンプレミス対応、OAuth トークン フェデレーション）
  - FR-002: データ取込み機能（MQTT → Event Hubs → LDP）
  - FR-003: アラート機能（異常検知、通知）
  - FR-004: マスタ管理機能（デバイス/ユーザー/組織/アラート設定/デバイス台帳/メール通知設定/アカウント/CSVインポート・エクスポート）
  - FR-005: 履歴機能（アラート履歴、メール通知履歴）
  - FR-006: 分析機能（ダッシュボード、対話型AI）
- 画面一覧（33画面）
- 画面遷移図・業務フロー図
- 実装優先順位（MVP → 拡張機能 → 高度な機能）
- 受け入れ条件
- 制約事項

### 🚀 非機能要件定義書
**ファイル**: `non-functional-requirements.md`
**目的**: システムが「どう動くか」を定義

**含まれる内容**:
- パフォーマンス要件
  - リアルタイムデータ処理（5分以内）
  - 大量データ処理（10,000デバイス以上対応）
  - Web画面応答性能（初期ロード3秒以内）
  - データベースクエリ性能
- 可用性要件
  - システム稼働率（99.5%以上）
  - データ整合性（損失率0.01%以下）
  - エラーハンドリング
  - 障害復旧（RTO: 4時間、RPO: 1時間）
- セキュリティ要件
  - 認証・認可（認証共通モジュール: Azure/AWS/オンプレミス対応、OAuth トークン フェデレーション、アクセス元IP制限）
  - データ保護（暗号化100%）
  - 入力検証（SQLインジェクション、XSS対策）
  - 外部通信制限（Private Linkによるセキュア通信）
  - ログ・監査
- スケーラビリティ要件
  - IoTデバイス数のスケーラビリティ（最大100,000デバイス）
  - データ量のスケーラビリティ（ペタバイト規模）
  - 同時接続ユーザー数（最大500ユーザー）
- 保守性要件
  - コード品質（テストカバレッジ80%以上）
  - ドキュメント整備
  - テスタビリティ
  - デプロイメント（30分以内）
- 監視・ログ要件
  - システム監視（5分間隔）
  - ログ管理（1年以上保持）
  - パフォーマンスメトリクス
  - アラート管理
- 互換性要件
  - ブラウザ対応（Chrome、Edge、Firefox、Safari最新2バージョン）
  - デバイス対応（デスクトップ、タブレット、モバイル）
- ユーザビリティ要件
  - UI/UX品質（タスク完了率95%以上）
  - レスポンシブデザイン
  - エラー処理
- アクセシビリティ要件
  - セマンティックHTML
  - キーボードナビゲーション
  - コントラスト（4.5:1以上）
  - フォームアクセシビリティ

### ⚙️ 技術要件定義書
**ファイル**: `technical-requirements.md`
**目的**: システムを「どう作るか」を定義

**含まれる内容**:
- Databricks Platform技術スタック
  - Lakeflow 宣言型パイプライン (LDP) - Python/SQL
  - App Compute - Flask Application
  - Unity Catalog - Delta Lake（ADLS連携）
  - 認証共通モジュール（Azure/AWS/オンプレミス対応）+ OAuth トークン フェデレーション
  - コンピューティングプレーン構成（サーバレス/クラシック）
- データベース技術スタック
  - Unity Catalog（センサーデータ、履歴データ）
  - MySQL互換データベース（マスタデータ、デバイスステータス、OLTP）
- Azure IoT Services技術スタック
  - Azure IoT Hubs（MQTT通信）
  - Event Hubs（Kafkaエンドポイント、Event Hubs Capture）
- ネットワーク構成
  - Private Link（Event Hub/ADLS/MySQL/Databricks UI API）
  - Azure DNS（Private Link名前解決）
- フロントエンド技術スタック
  - Flask 3.0+、Jinja2
  - HTML5、CSS3
  - 最小限のJavaScript
- バックエンド技術スタック
  - Python 3.10+
  - 必須パッケージ（Flask、databricks-sql-connector、PyMySQL等）
  - データベース接続（接続プーリング）
  - アラート処理（メール、Slack等）
- テスト要件
  - Pytest（ユニットテスト、カバレッジ80%以上）
  - Flask Test Client（統合テスト）
  - 手動テスト（UI確認、クロスブラウザ）
- ビルド・デプロイ要件
  - App Computeデプロイ（Git連携、アセットアップロード）
  - Databricks リソースデプロイ（Databricks CLI、アセットバンドル）
  - 環境管理（.env、Databricks環境変数）
- 開発環境要件
  - 必須ツール（VS Code、Git、Python 3.10+）
  - コード品質ツール（Pylint、Flake8、Black）
  - ローカル開発環境
- セキュリティ要件
  - 認証・認可（Entra ID連携、リバースプロキシヘッダ）
  - 入力検証（パラメータ化クエリ、XSS対策）
  - セキュアヘッダ
  - Private Linkによるセキュア通信
- パフォーマンス要件
  - データストア使い分け
  - 接続プーリング
  - キャッシュ戦略
- 監視・ログ要件
  - ログ出力（Unity Catalog、外部ログアダプター）
  - 監視（App Compute死活監視、メトリクス）
- アクセシビリティ要件
  - セマンティックHTML
  - フォームアクセシビリティ
- 拡張性要件
  - モジュール化（Flask Blueprint構成）

## 使用方法

### プロジェクト関係者向け
1. **プロジェクトマネージャー**: 機能要件定義書から開始し、実装優先順位を確認
2. **インフラエンジニア**: 技術要件定義書とインフラストラクチャドキュメントを参照
3. **データエンジニア**: Databricks関連の技術要件を確認（LDP、Unity Catalog）
4. **アプリケーションエンジニア**: Flask開発の技術要件、機能要件を参照
5. **QAエンジニア**: 非機能要件で品質基準を確認、受け入れ条件を参照

### 開発フェーズ別活用法
- **計画フェーズ**: 機能要件でスコープを確認、実装優先順位を決定
- **設計フェーズ**: 技術要件でアーキテクチャを決定、データベース設計を実施
- **開発フェーズ**: 全てのドキュメントで実装詳細を確認
- **テストフェーズ**: 非機能要件で品質基準を確認、受け入れ条件でチェック
- **デプロイフェーズ**: 技術要件のデプロイ手順を確認
- **運用フェーズ**: 監視・ログ要件を参照、パフォーマンス指標を確認

## 要件の管理と更新

### 更新プロセス
1. **変更要求**: GitHub Issuesで変更を申請
2. **影響分析**: 関連する他要件への影響を評価
3. **ドキュメント更新**: 変更をドキュメントに反映
4. **レビュー**: チームによるレビュー
5. **承認**: プロジェクトオーナーによる承認

### バージョン管理
- **メジャー変更** (v2.0): 機能の大幅追加・変更
- **マイナー変更** (v1.1): 既存機能の改善
- **パッチ変更** (v1.0.1): バグ修正・軽微な改善
