# Databricks IoTシステム 非機能要件定義書

**留意事項**:
- 指標を具体化できないものは参考値である旨を記載(指標値厳守ではない)
- 実装に直接関係ないものについてはセクションごとにその旨を明記（設計、実装へのinputとしては不要）

## 1. 概要

本ドキュメントは、Databricks IoTシステムの非機能要件を定義します。非機能要件は、システムが「どう動くか」を規定し、パフォーマンス、可用性、セキュリティ、スケーラビリティなどの品質属性を明確にします。

---

## 2. パフォーマンス要件

### NFR-PERF-001: リアルタイムデータ処理

**要件**:
- IoTデバイスからのデータをリアルタイムで処理すること
- ストリーミングデータの遅延を最小限に抑えること

**指標**:
- Event Hubs → LDP → シルバー層: **1分以内** 

**実装方針**:
- Lakeflow 宣言型パイプラインのストリーミング処理を活用
- Databricks Compute Clusterの自動スケーリング設定

---

### NFR-PERF-002: 大量データ処理

**要件**:
- 大量のセンサーデータを効率的に処理すること
- データ量増加に対してもパフォーマンスを維持すること

**指標**:
- 処理可能デバイス数: **10,000デバイス**
- 1デバイスあたりデータ送信頻度: **1分間隔**
- 1日あたりデータ処理量: **10GB程度**
- Unity Catalogデータ保持量: **ペタバイト規模対応**

**実装方針**:
- Delta Lakeの水平スケーリング活用
- パーティション戦略による性能最適化
- メダリオンアーキテクチャによる段階的処理

---

### NFR-PERF-003: Web画面応答性能

**要件**:
- 管理画面・ダッシュボードが快適に動作すること
- ユーザー操作に対して遅延なく応答すること

**指標**:
- 初期ページ読み込み時間: **3秒以内**
- 画面遷移時間: **1秒以内**
- API応答時間: **500ms以内**
- ダッシュボード表示時間: **10秒以内** (Databricks側の処理時間含む)

**実装方針**:
- Flask静的ファイル配信の最適化

- データベースクエリを効率的に実行すること

**指標**:

## 4. セキュリティ要件

### NFR-SEC-001: 認証・認可

**要件**:
- 適切な認証・認可機能を実装すること
  - **認証共通モジュール**: Azure/AWS/オンプレミス環境に対応可能な共通モジュールとして実装
  - **OAuth トークン フェデレーション**: Databricks接続時、ユーザー単位の認証とデータスコープ制御を実現
- ロールベースアクセス制御（RBAC）を実装すること
- ユーザーごとにアクセス権限を制御すること

**ロール定義**:
本システムでは4つのロールを定義し、各ロールに応じたアクセス権限を設定する。

| ロール         | 説明                                                                 | 権限レベル                               | 主な対象者               |
| -------------- | -------------------------------------------------------------------- | ---------------------------------------- | ------------------------ |
| システム保守者 | NSW内部の担当者。主に保守をスコープとする                            | 最上位権限（保守作業含む）               | NSW内部エンジニア        |
| 管理者         | システムの管理者。システム保守者以外（NSW外）で最大の権限を持つ      | 管理機能全般へのアクセス可能             | サービス提供企業の管理者 |
| 販社ユーザ     | システムの仲介業者。自社に紐づくデータに関連するタスクのみ操作できる | 自社データに関連する機能のみアクセス可能 | 販売代理店の担当者       |
| サービス利用者 | 本システムで管理するサービスを実際に利用しているユーザ。エンドユーザ | 参照機能とアラート設定のみアクセス可能   | エンドユーザ             |

**認証基盤**:
- **認証共通モジュール**: 「Azure環境(Easy Auth)」、「AWS環境(ALB＋cognito)」、「オンプレミス環境(自前認証)」の三パターンに対応可能
- **Azure環境での認証**:
  - インターネット → Entra ID でユーザ認証・トークン発行
  - アプリケーション → Easy Auth (Entra ID)でユーザアカウント制御
- **Databricks接続**: OAuth トークン フェデレーションによるユーザー単位認証
  - IdP発行トークン → Databricks Token Exchange → Unity Catalog接続
  - データスコープ制御を実現
- **Flaskアプリ**: リバースプロキシヘッダからユーザー情報を取得し、OAuth Token Exchangeを実行
- **アクセス元IP制限**:
  - Databricks ワークスペースへのアクセス元IP制限を実施
  - 公開フロントへのアクセス元IP制限を実施

**認証フロー（Azure環境の例）**:
1. ユーザがインターネット経由でEntra IDにアクセス
2. Entra IDがユーザ認証・トークン発行
3. リバースプロキシ（Easy Auth）がヘッダにユーザー識別情報とトークンを付与
4. Flaskアプリで受信したトークンを使用してOAuth Token Exchangeを実行
5. Unity Catalog接続時、OAuth トークン フェデレーションによりユーザー単位のデータスコープ制御を実現

**実装方針**:
- **認証共通モジュール**: AuthProviderパターンによる抽象化（Azure/AWS/オンプレミス対応）
- **OAuth Token Exchange**: Databricks SDK + databricks-sql-connector
- **リバースプロキシ**: 認証後、ヘッダにユーザー識別子・アクセストークンを付与
- **Flaskアプリ**: リバースプロキシヘッダからユーザー情報を取得し、Unity Catalog接続時にToken Exchangeを実行
- データベースまたはUnity Catalogにロール情報を保存
- Flask Blueprintレベルでの権限チェックデコレーター実装
- サービスプリンシパル権限とユーザー認可APIの使い分け
- 動的ビューによるデータアクセス制御

### NFR-SEC-006: ロールベースアクセス制御要件

**要件**:
- 機能レベルでロールに応じたアクセス制御を実装すること
- 権限のないユーザーが機能にアクセスできないようにすること
- 権限エラーを適切にハンドリングすること

**指標**:
- 参考値
- 権限チェック漏れ: **0件**
- 不正アクセス試行検出率: **100%**
- 権限エラーメッセージ表示率: **100%**

**実装方針**:

#### 画面レベルのアクセス制御
- Flask routeにデコレーター実装 (`@role_required('管理者', 'システム保守者')`)
- ログイン後のセッションにロール情報を保持
- 権限不足時は403 Forbiddenエラーと分かりやすいメッセージを表示

#### API レベルのアクセス制御
- 各APIエンドポイントでロールチェック実装
- CRUD操作ごとに必要なロールを定義
- 権限不足時はJSON形式でエラーレスポンス

#### 機能別アクセス制御マトリクス
詳細は `functional-requirements.md` の各機能要件のアクセス権限セクションを参照。

**主要な権限制限**:
- **デバイス台帳管理**: システム保守者のみアクセス可能
- **対話型AI機能**: ダッシュボード（FR-006-1）内に統合、ダッシュボード表示権限に準ずる

**権限エラー時の動作**:
- Web画面: 403エラーページ表示、ログイン画面へのリダイレクトリンク
- API: `{"error": "Forbidden", "message": "この操作を実行する権限がありません"}`
- ログ: 不正アクセス試行として監査ログに記録

### NFR-SEC-007: データスコープ制限要件

**要件**:
- 組織階層に基づいたデータアクセス範囲を制限すること
- すべてのユーザーに対して統一的なフィルタリングロジックを適用すること
- ロールはUI動線制御（ボタン・メニューの表示/非表示）のみに使用すること
- データアクセス範囲の違いは組織配置のみで実現すること

**指標**:
- 参考値
- データアクセス範囲チェック実施率: **100%**
- データ漏洩件数: **0件**
- 不正なデータアクセス試行検出率: **100%**

**実装方針**:

#### データベースレベルでの制限
**OLTP DB (MySQL互換)**:
- WHERE句による組織階層ベースのフィルタリング実装
- 全ユーザー共通: `WHERE organization_id IN (SELECT subsidiary_organization_id FROM organization_closure WHERE parent_organization_id = :current_user_organization_id)`
- ロールによる条件分岐は一切実装しない

**Unity Catalog**:
- 動的ビュー (Dynamic Views) を使用して組織階層ベースでデータをフィルタリング
- `current_user_organization_id()` 関数でユーザーの所属組織IDを取得
- `organization_closure` テーブルを使用して下位組織を取得
- **すべてのユーザーに対して同一のフィルタリングロジックを適用**（ロールによる条件分岐なし）
- ビュー例:
```sql
-- すべてのユーザーに対して組織階層ベースのフィルタリングを適用
-- ロールによる条件分岐は一切行わない
CREATE VIEW device_view AS
SELECT d.*
FROM devices d
WHERE d.delete_flag = FALSE
  AND EXISTS (
    -- 組織階層によるフィルタリング（全ユーザー共通）
    SELECT 1
    FROM organization_closure oc
    WHERE oc.parent_organization_id = current_user_organization_id()
      AND oc.subsidiary_organization_id = d.organization_id
  );
```

#### アプリケーションレベルでの制限
- セッションに保存されたユーザー情報（ロール、所属組織ID）を使用
- データ取得クエリに自動的にスコープフィルターを追加
- CSVエクスポート時もスコープ制限を適用

**最重要原則**:
- **すべてのロールで完全に同一のフィルタリングロジックを適用**
- **ロールによるデータ制限は実施しない（画面操作上の制限のみに使用）**
- データアクセス範囲の違いは、ユーザーが所属する**組織の配置**で決定

**データアクセス範囲の決定方法（全ユーザー共通）**:
1. ログインユーザーの `organization_id` を取得
2. `organization_closure` テーブルで `parent_organization_id = ユーザーのorganization_id` のレコードを検索
3. 取得した `subsidiary_organization_id` のリストがアクセス可能な組織リスト
4. このリストに該当する組織のデータのみを表示

**組織構造の設計例**:
- すべてのユーザーは所属組織とその下位組織のデータにアクセス可能
- 組織階層の上位に配置されたユーザーほど、より多くのデータにアクセス可能
- 組織階層の下位に配置されたユーザーは、限定されたデータのみアクセス可能

**データスコープ違反時の動作**:
- 該当データが存在しない（404 Not Found）として処理
- 監査ログに不正アクセス試行として記録
- ユーザーには「データが見つかりません」というメッセージを表示

### NFR-SEC-008: 自前認証時のパスワードポリシー（オンプレミス環境）

**要件**:
オンプレミス環境で自前認証を使用する場合、以下のパスワードポリシーを適用すること

**パスワード複雑度**:

| 項目 | 要件 |
|------|------|
| 最小文字数 | 8文字以上 |
| 最大文字数 | 128文字以下 |
| 必須文字種 | 大文字・小文字・数字・記号のうち3種類以上 |
| 禁止パターン | ユーザーID、メールアドレスの一部を含まない |

**パスワード有効期限**:

| 項目 | 設定値 |
|------|--------|
| 有効期限 | 90日（設定可能） |
| 期限切れ警告 | 14日前から通知 |
| 履歴保持 | 過去5世代は再利用禁止 |

**アカウントロック要件**:

| 項目 | 設定値 |
|------|--------|
| 連続失敗回数 | 5回 |
| ロック期間 | 30分（自動解除） |
| 管理者による手動解除 | 可能 |

**パスワードリセット要件**:

| 項目 | 設定値 |
|------|--------|
| リセットトークン有効期限 | 1時間 |
| 通知方法 | メール送信 |
| リセット後の処理 | 即時ログアウト、再認証必須 |

**実装方針**:
- bcryptによるパスワードハッシュ化（コストファクター12）
- 秘密の質問は使用しない（セキュリティ上の理由）
- 多要素認証（MFA）は将来の拡張として検討
- 詳細は[認証仕様書](../03-features/common/authentication-specification.md)を参照

### NFR-SCALE-001: IoTデバイス数のスケーラビリティ
-**実装には関与しない要件事項**

**要件**:
