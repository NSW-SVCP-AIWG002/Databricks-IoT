## 📋 概要

このドキュメントは、タスク管理アプリケーションの結合テスト（Integration Test）を実装する際のガイドラインです。複数のコンポーネント・レイヤー間の連携が正しく動作することを検証します。

**テストケースの書き方、ベストプラクティス、サンプルコードについては [テスト実装ルール](../../00-rules/test-rule.md) を参照してください。**

---

## 📖 このドキュメントの使い方

このガイドは**汎用的なテンプレート・リファレンス**として活用してください。

### 利用フロー

```
integration-test-guide.md (99項目の汎用チェックリスト)
       ↓ 参照
integration-test/task-management-spec.md (タスク管理に該当する15項目を詳細化)
       ↓ 実装
test/integration/tasks/create-task-flow.test.ts (実際のテストコード)
```

### 具体的な使い方

1. **新機能実装時**: このガイドを開き、結合テスト項目一覧を確認
2. **項目選択**: APIフロー、認証フロー、データフローなどから該当項目をピックアップ
3. **仕様書作成**: `integration-test/<機能名>-spec.md` に機能固有の結合テストシナリオを記載
4. **実装ガイド参照**: [テスト実装ルール](../../00-rules/test-rule.md) のサンプルコードを参考に実装
5. **テスト実装**: 実際の結合テストコード（`.test.ts`）を作成

### 注意点

- このガイドは**汎用的な連携観点**のみを記載（例: 「全レイヤーが正しく動作すること」）
- **機能固有のシナリオ**は `integration-test/` に記載（例: 「契約作成→承認→通知の一連フロー」）
- **テストコードの実装方法**は [テスト実装ルール](../../00-rules/test-rule.md) を参照

## 🎯 結合テストの目的とスコープ

### 目的

- 複数のモジュール・レイヤー間の連携が正しく動作することを保証
- インターフェースの整合性確認
- データフローの妥当性検証
- システム全体の動作確認（E2Eの一歩手前）

### スコープ

- **対象**: 複数のレイヤーを跨ぐ処理（Handler → UseCase → Repository → DB）
- **外部依存**: 一部をモック化（外部API、メール送信など）
- **実行速度**: 中速（全体で2分以内）
- **カバレッジ目標**: 最低90%、推奨95%以上

### 単体テストとの違い

| 項目 | 単体テスト | 結合テスト |
| ------| ---------| ---------|
| テスト範囲 | 単一の関数・モジュール | 複数のレイヤー・モジュール |
| データベース | モック化 | 実DBまたはテスト用DB |
| 外部API | モック化 | モック化（一部は実接続） |
| 実行時間 | 高速（< 30秒） | 中速（< 2分） |
| 目的 | 個別機能の正確性 | 連携の正確性 |

## 🛠️ テスト環境・ツール

### バックエンド（task-api）

- **テストランナー**: Spring Boot (@SpringBootTest + TestRestTemplate)
- **HTTPクライアント**: TestRestTemplate
- **データベース**: テスト用DB（独立したスキーマ）
- **モック**: 外部APIのみモック化
- **環境変数**: `AIOPS_ENV=test`

### フロントエンド（task-web）

- **テストランナー**: Next.js (Jest + React Testing Library)
- **APIモック**: MSW (Mock Service Worker)
- **コンポーネントテスト**: React Testing Library
- **ルーティング**: Next.js Router のモック
