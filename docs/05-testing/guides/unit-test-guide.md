## 📋 概要

このドキュメントは、タスク管理アプリケーションの単体テスト（Unit Test）の仕様書を作成する際のガイドラインです。

[単体テスト観点チェックリスト](./unit-testing-checklist.md) から選択したテスト項目を、どのような形式で仕様書に記載するかの**形式例**を提供します。

**テストケースの書き方、ベストプラクティス、サンプルコードについては [テスト実装ルール](../../00-rules/test-rule.md) を参照してください。**

---

## 📖 このドキュメントの使い方

このガイドは**汎用的なテンプレート・リファレンス**として活用してください。

### 利用フロー

```
unit-testing-checklist.md (汎用的な観点チェックリスト)
       ↓ 参照・選択
unit-test-guide.md (選択した項目の仕様書作成形式例)
       ↓ 参照
unit-test/task-management-spec.md (機能固有の詳細仕様書)
       ↓ 実装
app/task-api/src/repository/tasks/create-task.test.ts (実際のテストコード)
```

### 具体的な使い方

1. **新機能実装時**: [単体テスト観点チェックリスト](./unit-testing-checklist.md) を開き、関連する観点を確認
2. **項目選択**: チェックリストから該当する機能に必要な項目をピックアップ
3. **形式確認**: このガイドの「仕様書作成の形式例」を参照し、テストケースの書き方を確認
4. **仕様書作成**: `unit-test/<機能名>-spec.md` に機能固有のテストケースを記載
5. **実装ガイド参照**: [テスト実装ルール](../../00-rules/test-rule.md) のサンプルコードを参考に実装
6. **テスト実装**: 実際のテストコード（`.test.ts`）を作成

### 注意点

- [単体テスト観点チェックリスト](./unit-testing-checklist.md) は**汎用的な観点**を網羅的に記載
- このガイドは**仕様書作成の形式例**を提供（テーブル形式、記載内容の例）
- **機能固有の詳細**は `unit-test/` に記載（例: 「契約番号 C-2025-001 で作成されること」）
- **テストコードの実装方法**は [テスト実装ルール](../../00-rules/test-rule.md) を参照

## 🎯 単体テストの目的

- 個別の関数・モジュール・コンポーネントが仕様通りに動作することを保証
- バグの早期発見と修正
- リファクタリング時の安全性確保
- コードの品質向上

## 📊 仕様書作成の形式例

このセクションでは、[単体テスト観点チェックリスト](./unit-testing-checklist.md) から選択した項目を、どのような形式で仕様書に記載するかの例を示します。

### 形式の基本構成

各テストケースは以下の情報を含むテーブル形式で記載します：

- **No.**: 連番
- **分類**: [単体テスト観点チェックリスト](./unit-testing-checklist.md) の小分類（例: 1.1 検索条件、2.1 基本入力チェック）
- **観点**: テストする観点（作成、取得、更新、削除など）
- **内容**: 具体的なテストの内容
- **備考**: 追加の注意事項や補足情報

### テストケース記載例

以下は、[単体テスト観点チェックリスト](./unit-testing-checklist.md) から項目を選択し、仕様書に記載する際の形式例です。

#### 🔍 1. 検索機能

| No. | 分類           | 観点       | 内容                                         | 備考         |
| --- | -------------- | ---------- | -------------------------------------------- | ------------ |
| 1   | 1.1 検索条件   | 単一条件   | 単一条件での検索が正しく動作すること         |              |
| 2   | 1.1 検索条件   | 複数条件   | AND/OR 条件での検索が正しく動作すること      |              |
| 3   | 1.1 検索条件   | 未入力     | 条件未入力時に全件検索されること             |              |
| 4   | 1.2 検索結果   | 0 件       | 0 件の場合に適切なメッセージが表示されること |              |
| 5   | 1.2 検索結果   | ページング | ページング処理が正しく動作すること           | 境界値の確認 |

#### ✏️ 2. 入力系

| No. | 分類                 | 観点         | 内容                                       | 備考 |
| --- | -------------------- | ------------ | ------------------------------------------ | ---- |
| 6   | 2.1 基本入力チェック | 必須項目     | 必須項目が未入力の場合エラーとなること     |      |
| 7   | 2.1 基本入力チェック | 最大文字数   | 最大文字数を超えた場合エラーとなること     |      |
| 8   | 2.2 文字種制限       | メール形式   | メールアドレス形式のチェックが動作すること |      |
| 9   | 2.3 日付・時刻入力   | 存在しない日 | 2 月 30 日などでエラーとなること           |      |
| 10  | 2.3 日付・時刻入力   | 前後関係     | 開始日 > 終了日の場合エラーとなること      |      |

#### 🖥️ 3. 表示系

| No. | 分類                   | 観点             | 内容                                     | 備考          |
| --- | ---------------------- | ---------------- | ---------------------------------------- | ------------- |
| 11  | 3.1 データ表示         | 0 件表示         | 0 件時に適切なメッセージが表示されること |               |
| 12  | 3.1 データ表示         | 長文データ       | 長文データが適切に折り返し表示されること | ellipsis など |
| 13  | 3.2 日付・時刻表示     | 日付フォーマット | 日付が YYYY/MM/DD 形式で表示されること   |               |
| 14  | 3.3 個人情報・機密情報 | パスワードマスク | パスワードがマスク表示されること         |               |

#### 🗄️ 6. データベース操作

| No. | 分類         | 観点       | 内容                                     | 備考                           |
| --- | ------------ | ---------- | ---------------------------------------- | ------------------------------ |
| 15  | 6.1 登録処理 | 正常登録   | データが正しく DB に保存されること       | デフォルト値の確認も含む       |
| 16  | 6.1 登録処理 | 重複データ | ユニーク制約違反時にエラーが発生すること |                                |
| 17  | 6.2 更新処理 | 正常更新   | データが正しく更新されること             | 部分更新も確認                 |
| 18  | 6.2 更新処理 | 排他制御   | 同時更新時に適切に制御されること         | 楽観的ロック/悲観的ロック      |
| 19  | 6.3 削除処理 | 正常削除   | データが正しく削除されること             | 削除後に取得できないことを確認 |

#### ⚠️ 10. エラー処理

| No. | 分類                 | 観点           | 内容                                      | 備考                 |
| --- | -------------------- | -------------- | ----------------------------------------- | -------------------- |
| 20  | 10.1 エラーメッセージ | エラー箇所明示 | エラー箇所がハイライトされること          | 該当項目の赤枠表示等 |
| 21  | 10.1 エラーメッセージ | 複数エラー     | 複数エラー時に全エラーが表示されること    |                      |
| 22  | 10.2 例外処理        | 404 エラー     | 存在しないリソースで 404 エラーとなること |                      |
| 23  | 10.2 例外処理        | ネットワークエラー | ネットワークエラーが適切に処理されること  |                      |

#### ♿ 15. アクセシビリティ

| No. | 分類               | 観点       | 内容                               | 備考 |
| --- | ------------------ | ---------- | ---------------------------------- | ---- |
| 24  | 15 アクセシビリティ | role 属性  | 適切な role 属性が設定されていること |      |
| 25  | 15 アクセシビリティ | Tab 移動   | Tab キーでフォーカス移動できること |      |
| 26  | 15 アクセシビリティ | Enter 送信 | Enter キーでフォーム送信できること |      |

**※ 上記は記載例です。実際の仕様書では、プロジェクトの機能に応じて [単体テスト観点チェックリスト](./unit-testing-checklist.md) から必要な項目を選択し、同様の形式で記載してください。**
