## 📋 概要

このドキュメントは、プロジェクトにおけるテストカバレッジ戦略と品質メトリクスを定義します。カバレッジ目標値の設定根拠、測定方法、品質 KPI を明確にし、プロジェクトの品質を定量的に管理するための基盤となります。

**関連ドキュメント**: [テスト戦略・方針](./test-strategy.md) - テストの全体的な戦略と実装方針

## 📊 テストカバレッジ戦略

### カバレッジの定義

本プロジェクトでは以下のカバレッジ指標を採用し、品質を保証します。

| カバレッジ種別                 | テストレベル                    | 最低目標値            | 推奨目標値            | 測定方法                                                    | 説明                                                                          |
| ------------------------------ | ------------------------------- | --------------------- | --------------------- | ----------------------------------------------------------- | ----------------------------------------------------------------------------- |
| **コードカバレッジ**           | 単体テスト（主）                | 70%以上               | 80%以上               | バックエンド: JaCoCo (Gradle)<br>フロントエンド: Jest (Bun) | Line, Branch, Function の各カバレッジを測定                                   |
| **機能カバレッジ**             | 単体 + 統合 + E2E（全レベル）   | 80%以上               | 90%以上               | 全機能に対するテストケース作成率                            | 主要機能 100%、副次機能 60%以上をカバー                                       |
| **統合ポイントカバレッジ**     | 統合テスト                      | 90%以上               | 95%以上               | 全 API エンドポイントのテストケース作成率                   | 全公開 API エンドポイント + 主要な内部 API の結合テストを保証                 |
| **クリティカルパスカバレッジ** | E2E テスト                      | 100%（必達）          | 100%                  | クリティカルユーザーフローのテストケース作成率              | 全てのクリティカルユーザーフローの E2E テストが存在することを保証（妥協不可） |
| **画面カバレッジ**             | E2E テスト（主）+ UI 単体テスト | 80%以上               | 90%以上               | 全画面に対するテストケース作成率                            | 主要画面・公開画面 100%、管理画面 60%以上をカバー                             |
| **ブラウザカバレッジ**         | E2E テスト                      | 主要 3 ブラウザ       | 主要 3 ブラウザ       | Chrome、Edge、Safari 等                                     | クロスブラウザ互換性の確認                                                    |
| **デバイスカバレッジ**         | E2E テスト                      | デスクトップ/モバイル | デスクトップ/モバイル | レスポンシブデザイン対応確認                                | 各デバイスでの表示・操作性を検証                                              |

### カバレッジ目標値の設定根拠

本プロジェクトでは、実案件における現実的なリソース配分と品質保証のバランスを考慮し、以下の根拠に基づいてカバレッジ目標値を設定しています。

#### 1. **コードカバレッジ: 最低 70%、推奨 80%**

**業界標準との比較**:

- Google、Microsoft などの大手企業: 70-85%
- エンタープライズアプリケーション: 60-75%
- スタートアップ/小規模プロジェクト: 50-70%

**70%が最低値である理由**:

- **主要ロジックのカバー**: 70%で、ビジネスロジック、データアクセス、重要な処理パスをほぼカバーできる
- **バグリスクの高い部分を優先**: 残り 30%は、簡単な getter/setter、定型的なボイラープレート、UI の細かいスタイリングなど、バグリスクが相対的に低い部分
- **ROI（投資対効果）**: 70%超えを目指すと、テストのためのテストを書くことになり、開発時間とメンテナンスコストが急増
- **実績**: 多くの商用プロジェクトで 70%以上が品質保証の最低ラインとして採用されている

**80%が推奨値である理由**:

- **品質への余裕**: 80%であれば、エッジケースや異常系もある程度カバーできる
- **リファクタリングの安心感**: カバレッジが高いほど、コード変更時の影響範囲を早期に検出できる
- **ステークホルダーへの説明責任**: 外部への品質報告において、80%は「優秀」と評価される水準

#### 2. **機能カバレッジ: 最低 80%、推奨 90%**

**80%が最低値である理由**:

- **パレートの法則**: ユーザーの 80%が使用するのは全機能の 20%
- **優先順位づけ**: 全機能に均等にリソースを投じるのではなく、主要機能（タスク管理、認証など）に 100%を達成し、副次機能（設定変更、プロフィール編集など）は 60%程度で許容
- **リソース制約**: 全機能 100%を目指すと、低頻度の管理画面や例外的なフローにも同等のテストを書くことになり、ROI が悪化
- **実案件での実績**: エンタープライズアプリケーションでは、主要機能 100%+全体 80%が一般的

**90%が推奨値である理由**:

- **包括的な品質保証**: 90%であれば、ほとんどの機能要件をカバーでき、想定外の不具合を大幅に削減
- **ユーザー満足度**: 副次機能でも一定のテストがあることで、ユーザー体験が安定

#### 3. **統合ポイントカバレッジ: 最低 90%、推奨 95%**

**90%が最低値である理由**:

- **公開 API は 100%必須**: 外部に公開する API エンドポイントは、利用者への影響が大きいため 100%カバー必須
- **内部 API の優先順位**: 内部 API は、重要度に応じて選別。頻繁に使用される内部 API は 100%、低頻度のものは省略可能
- **パラメータ組み合わせの現実性**: 全エンドポイント × 全パラメータ組み合わせのテストは非現実的。代表的なパターンに絞る
- **実案件での実績**: REST API を持つ Web アプリケーションでは、公開 API 100% + 主要内部 API = 全体 90%が現実的

**95%が推奨値である理由**:

- **API 品質への信頼**: 95%であれば、ほぼ全ての API エンドポイントがテストされており、外部連携時のトラブルを最小化
- **ドキュメントとの整合性**: API ドキュメントに記載された全エンドポイントがテスト済みであることを保証

#### 4. **クリティカルパスカバレッジ: 100%（必達・妥協不可）**

**100%が絶対条件である理由**:

- **ビジネス継続性**: クリティカルパスの不具合は、サービス停止・売上損失・業務不能に直結
- **リリースブロッカー**: クリティカルパスのテストが失敗した場合、リリースを中止すべき重要度
- **ユーザー信頼**: ログインできない、データが登録できないなどの問題は、ユーザーの信頼を著しく損なう
- **代替不可**: クリティカルパスは他の手段で代替できない操作であり、100%の動作保証が必須
- **実現可能性**: クリティカルパスの定義を厳選すれば（6 つの判断指標で「クリティカル」と判定されたもののみ）、100%達成は現実的
- **業界標準**: どの商用アプリケーションでも、クリティカルパスの E2E テストは 100%が基本

#### 5. **画面カバレッジ: 最低 80%、推奨 90%**

**80%が最低値である理由**:

- **主要画面・公開画面は 100%**: ユーザーが日常的に利用する画面（タスク一覧、ログインなど）と、公開画面は 100%カバー必須
- **管理画面の優先順位**: 管理画面は利用頻度が低く、影響範囲も限定的。60%程度で許容
- **E2E テストのコスト**: 画面テストは実行時間が長く、メンテナンスコストも高い。全画面 100%は非現実的
- **実案件での実績**: B2C 向けサービスでは、ユーザー向け画面 100% + 管理画面 60% = 全体 80%が一般的

**90%が推奨値である理由**:

- **ユーザー体験の一貫性**: 管理画面でも主要な操作（ユーザー登録、データ更新など）がテスト済みであれば、安心してリリース可能
- **リグレッション防止**: 画面の変更は他の画面に影響を与えることがあり、90%のカバレッジでリグレッションを早期検出

#### まとめ: 最低値と推奨値の使い分け

| フェーズ             | 採用する目標値 | 説明                                                 |
| -------------------- | -------------- | ---------------------------------------------------- |
| **MVP/初期リリース** | 最低目標値     | リソースが限られている場合、最低値を死守しリリース   |
| **安定運用フェーズ** | 推奨目標値     | プロダクトが成熟し、品質への投資ができる段階で推奨値 |
| **エンタープライズ** | 推奨目標値     | 大規模・長期運用が前提の場合、推奨値を標準とする     |

**重要**: クリティカルパスカバレッジ 100%だけは、どのフェーズでも妥協不可です。

## 📈 品質メトリクス・KPI

### テスト品質指標

| メトリクス                   | 目標値             | 測定ツール                                                  |
| ---------------------------- | ------------------ | ----------------------------------------------------------- |
| コードカバレッジ（全体）     | 最低 70%、推奨 80% | バックエンド: JaCoCo (Gradle)<br>フロントエンド: Jest (Bun) |
| 単体テスト成功率             | 100%               | CI/CD パイプライン                                          |
| 統合テスト成功率             | 100%               | CI/CD パイプライン                                          |
| E2E テスト成功率             | 100%               | CI/CD パイプライン                                          |
| テスト実行時間（単体テスト） | < 40 秒            | CI/CD パイプライン                                          |
| テスト実行時間（統合テスト） | < 3 分             | CI/CD パイプライン                                          |
| テスト実行時間（E2E）        | < 6 分             | CI/CD パイプライン                                          |

### バグ検出メトリクス

| 指標                   | 目標      | 効果                 |
| ---------------------- | --------- | -------------------- |
| 本番環境でのバグ発生率 | < 3 件/月 | テストによる事前検出 |
| バグ修正リードタイム   | < 24 時間 | テストによる早期発見 |
| リグレッションバグ     | 0 件      | 自動テストによる防止 |
