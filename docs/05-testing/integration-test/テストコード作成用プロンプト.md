# 結合テストコード作成用プロンプト

以下の `<機能名>` を対象機能に差し替えて使用してください。

---

````
以下の手順に従い、<機能名> の結合テストコードを作成してください。

## 入力ファイル（必ず読み込むこと）

1. UI設計書: docs/03-features/flask-app/<機能名>/ui-specification.md
2. 機能設計書: docs/03-features/flask-app/<機能名>/workflow-specification.md
3. 結合テスト観点表: docs/05-testing/integration-test/integration-test-perspectives.md
4. 結合テスト実装ガイド: docs/05-testing/integration-test/integration-test-guide.md

※ 上記パスにファイルがない場合は、サブフォルダ（書面レビュー対象資料/ 等）内を探索すること。

## 作業手順

1. UI設計書と機能設計書を読み、機能の振る舞い（APIエンドポイント、DB操作、画面遷移等）を把握する
2. 結合テスト観点表を読み、該当する機能に必要な観点を選択する
3. 結合テスト実装ガイドの「テストコード実装例」を参照し、コードの書き方を確認する
4. 設計書に記載された具体的な値（エンドポイントURL、HTTPステータスコード、DB項目、バリデーション条件等）をテストコードに反映する

## 出力先

tests/integration/<機能名>/test_<ファイル名>_routes.py

※ サブディレクトリが存在しない場合は新規作成すること。
※ フィクスチャは tests/integration/<機能名>/conftest.py に記載する。
  autouse フィクスチャは必ずサブディレクトリ側の conftest.py に置き、
  tests/integration/conftest.py（親）には記載しないこと。

## 出力形式

pytestのテストコードとして出力すること。以下のルールに従う：
- クラスのdocstringに、対応する観点表のセクション番号を記載する
- 各テストメソッドのdocstringに、対応する観点表の項目番号と概要を記載する
- AAAパターン（Arrange-Act-Assert）に従う
- @pytest.mark.integration マーカーを付与する
- Flask test_client + Azure Database for MySQL（実DB）での検証が前提。外部API（Databricks等）はモック化対象として扱う
- 具体的なコードパターンは結合テスト実装ガイドの「テストコード実装例」を参照する

## 注意事項

- 観点表のすべての項目を機械的に適用しないこと。設計書に該当する機能・項目がある観点のみ選択する
- 設計書に明記されていない挙動は、テストコード内にコメントで「# TODO: 設計書に記載なし、要確認」と記載する
- **テストファイルがすでに存在する場合は、ファイル全体を上書きせず、変更・追加が必要なテストケースのみを差分で更新すること**
  - 既存のテストケースで変更不要なものは手を加えない
  - 新規追加するテストケースはファイル末尾または該当クラス内の適切な位置に追記する
  - 既存テストの削除・リネームは、設計書上の仕様変更が明示されている場合のみ行う
- **fixture のteardown はロールバックで行うこと**
  - function スコープの fixture では `yield` 後に `db.session.rollback()` を使用する
  - session スコープの fixture（マスタデータ等、複数テストで共有するもの）のみ、FK制約の順序に注意しながら明示的削除（`db.session.query(...).delete()` + `db.session.commit()`）を使用する
````
