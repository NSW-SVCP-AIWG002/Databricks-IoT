## 3. 共通ヘッダー

### 3.1 必須ヘッダー

すべてのリクエストには以下のヘッダーを含める必要があります：

```http
Authorization: Bearer {access_token}
Content-Type: application/json
Accept: application/json
```

#### 各ヘッダーの説明

| ヘッダー名 | 役割 | 説明 |
|-----------|------|------|
| Authorization | 認証 | Bearer Token（JWT）による認証。ログイン時に発行されるアクセストークンを指定 |
| Content-Type | 送信データ形式 | クライアントがサーバーに送るデータの形式を指定（リクエストボディ） |
| Accept | 受信データ形式 | サーバーからクライアントに返してほしいデータの形式を指定（レスポンスボディ） |

**Content-TypeとAcceptの違い:**
- **Content-Type**: 「送る」データの形式（クライアント → サーバー）
- **Accept**: 「受け取る」データの形式（サーバー → クライアント）
- 方向が異なるため、両方の指定が必要

### 3.2 オプションヘッダー

```http
X-Request-ID: {request_id}  # リクエストトレーシング用のユニークID
Accept-Language: ja         # 応答言語（デフォルト: ja）
```

## 4. 共通HTTPステータスコード

| コード | 説明 | 使用場面 |
|--------|------|---------|
| 200 | OK | 正常処理（取得・更新成功） |
| 201 | Created | リソース作成成功 |
| 204 | No Content | 正常処理（レスポンスボディなし、削除成功など） |
| 400 | Bad Request | リクエスト不正（パラメータエラー、バリデーションエラー） |
| 401 | Unauthorized | 認証エラー（トークンなし、期限切れ） |
| 403 | Forbidden | 権限不足 |
| 404 | Not Found | リソース未検出 |
| 409 | Conflict | 競合エラー（重複登録など） |
| 422 | Unprocessable Entity | バリデーションエラー |
| 429 | Too Many Requests | レート制限超過 |
| 500 | Internal Server Error | サーバーエラー |
| 502 | Bad Gateway | 外部API連携エラー |
| 503 | Service Unavailable | メンテナンス中 |

## 6. レスポンス形式

### 6.1 成功時のレスポンス

```json
{
  "success": true,
  "data": {
    // エンドポイント固有のデータ
  },
  "timestamp": "2025-11-01T10:00:00+09:00"
}
```

#### フィールド説明

| フィールド | 型 | 説明 |
|-----------|-----|------|
| success | boolean | 処理成功フラグ（常にtrue） |
| data | object | エンドポイント固有のレスポンスデータ |
| timestamp | string | レスポンス生成日時（ISO 8601形式、JST） |

### 6.2 エラー時のレスポンス

```json
{
  "success": false,
  "error": {
    "code": "ERROR_CODE",
    "message": "エラーメッセージ",
    "details": {
      // エラー詳細（任意）
    }
  },
  "timestamp": "2025-11-01T10:00:00+09:00"
}
```

#### フィールド説明

| フィールド | 型 | 説明 |
|-----------|-----|------|
| success | boolean | 処理成功フラグ（常にfalse） |
| error | object | エラー情報 |
| error.code | string | エラーコード（共通エラーコード参照） |
| error.message | string | ユーザー向けエラーメッセージ |
| error.details | object | エラー詳細情報（任意、バリデーションエラー時など） |
| timestamp | string | レスポンス生成日時（ISO 8601形式、JST） |

## 7. エラーハンドリング方針

すべてのAPIエンドポイントで共通するエラーハンドリングの方針を定義します。

### 7.1 エラー分類とHTTPステータス

| エラー分類 | 説明 | 対応方針 | HTTPステータス | トランザクション |
|-----------|------|---------|--------------|----------------|
| バリデーションエラー | 入力パラメータの形式・値が不正 | エラーレスポンス返却 | 400, 422 | 開始前 |
| 認証エラー | トークンなし、期限切れ、不正 | エラーレスポンス返却 | 401 | 開始前 |
| 認可エラー | 権限不足 | エラーレスポンス返却 | 403 | 開始前 |
| リソース不在エラー | 対象データが存在しない | エラーレスポンス返却 | 404 | 読み取りのみ |
| 競合エラー | データの重複・競合 | エラーレスポンス返却 | 409 | 開始後（ロールバック） |
| レート制限超過 | API呼び出し制限超過 | エラーレスポンス返却 | 429 | 開始前 |
| データベースエラー | DB接続失敗、SQL実行失敗 | ロールバック後エラーレスポンス返却 | 500 | 開始後（ロールバック） |
| 外部API連携エラー | 外部サービスとの連携失敗 | ロールバック後エラーレスポンス返却 | 502 | 開始後（ロールバック） |
| タイムアウト | 処理時間超過 | ロールバック後エラーレスポンス返却 | 504 | 開始後（ロールバック） |

**注:** トランザクション管理の詳細は[トランザクション管理](#15-トランザクション管理)セクションを参照してください。

### 7.2 リトライ方針

| エラー種別 | リトライ | 最大回数 | 間隔 | 備考 |
|-----------|---------|---------|------|------|
| バリデーションエラー | ❌ しない | - | - | クライアント側で修正が必要 |
| 認証エラー | ❌ しない | - | - | トークン再取得が必要 |
| 認可エラー | ❌ しない | - | - | 権限不足のため |
| リソース不在エラー | ❌ しない | - | - | データが存在しない |
| 競合エラー | ❌ しない | - | - | クライアント側で再試行を判断 |
| データベースエラー | ❌ しない | - | - | トランザクションの整合性保持のため |
| 外部API連携エラー | ⚠️ API依存 | - | - | 各API処理設計書で定義 |
| タイムアウト | ❌ しない | - | - | 処理の冪等性が保証されない場合 |

**注:** リトライが必要な場合は、個別のAPI処理設計書で詳細を定義してください。

### 7.3 エラーログ出力

エラー発生時は、以下の情報をログに出力します。

**必須項目:**
- リクエストID（トレーシング用）
- タイムスタンプ
- エラーコード
- HTTPステータス
- エンドポイント
- HTTPメソッド
- ユーザーID（認証済みの場合）
- 契約ID（認証済みの場合）

**任意項目:**
- エラーメッセージ
- スタックトレース（500エラーのみ）
- クエリパラメータ
- リクエストボディ（機密情報を除く）

**注:** ログ出力の詳細は[ログ出力ポリシー](#16-ログ出力ポリシー)セクションを参照してください。

## 8. 認証・認可

### 8.1 認証方式

| 項目 | 内容 |
|------|------|
| 認証基盤 | Azure AD (Entra ID) |
| 認証方式 | Bearer Token (JWT) |
| トークン有効期限 | 1時間 |
| リフレッシュトークン有効期限 | 7日間 |
| 認証ヘッダー | `Authorization: Bearer {access_token}` |

### 8.2 認証フロー

1. ユーザーがAzure AD (Entra ID)でログイン
2. 認証成功後、アクセストークン（JWT）を取得
3. 以降のAPIリクエストで、Authorizationヘッダーにトークンを付与
4. サーバー側でトークンを検証し、ユーザー情報と権限を確認
5. トークン期限切れ時は、リフレッシュトークンで更新

### 8.3 JWTトークン構造

```json
{
  "sub": "usr_001",                    // ユーザーID
  "name": "山田太郎",                   // ユーザー名
  "email": "yamada@example.com",       // メールアドレス
  "contract_id": "cnt_001",            // 契約ID
  "role": "system_admin",              // ロール
  "exp": 1730390400,                   // トークン有効期限（Unixタイムスタンプ）
  "iat": 1730386800                    // トークン発行日時（Unixタイムスタンプ）
}
```

### 8.4 ロール定義

システムには4つのロールが定義されています：

| ロール | 説明 | 権限範囲 |
|--------|------|---------|
| `super_admin` | 運営元管理者 | サービス管理の管理権限。契約に関する機能にアクセス可能 |
| `system_admin` | システム管理者 | 設定契約内の運用管理の権限。運用管理機能の全機能にアクセス可能 |
| `operation_admin` | 運用管理者 | 自契約内の運用管理の権限。運用管理機能の全機能にアクセス可能 |
| `user` | 一般ユーザー | 自契約内のRAG機能の利用権限。サービス管理・運用管理は全権限利用不可であり、現場APPのみ利用可能 |

### 8.5 ロールチェック

- すべてのAPIエンドポイントで、リクエスト時にJWTトークンから権限を確認
- 必要な権限がない場合は `403 Forbidden` を返却

### 8.6 権限エラー時のレスポンス

```json
{
  "success": false,
  "error": {
    "code": "PERMISSION_DENIED",
    "message": "この操作を実行する権限がありません",
    "details": {
      "required_role": "system_admin",
      "current_role": "user"
    }
  },
  "timestamp": "2025-11-01T10:00:00+09:00"
}
```

**HTTPステータス:** 403 Forbidden

## 9. クライアントサイドキャッシュ戦略

### 9.1 概要

**本プロジェクトの一覧APIでは、クライアント側でデータをキャッシュし、検索・フィルタリング・ソート・ページネーションをクライアント側で処理することを基本方針とします。**

**現状の想定データ件数: 各一覧で最大500件程度**

この件数であれば、クライアント側キャッシュで以下の利点が得られます：
- APIコール数の削減によるサーバー負荷軽減
- ページング・ソート・検索時の即座なレスポンス
- ユーザー体験（UX）の向上

**将来的なスケーラビリティ:**

データ件数が増加し、パフォーマンス問題が発生した場合は、個別のAPIごとにサーバーサイドページネーションへの移行を検討します。移行は比較的容易に実施可能です。詳細は[API実装ベストプラクティス](./implementation-best-practices.md#戦略切り替え可能な実装パターン)を参照してください。

### 9.2 データ件数に応じた推奨戦略

| データ件数（1契約あたり） | 推奨戦略 | 理由 |
|-------------------------|---------|------|
| ~500件（現状） | ✅ クライアント側キャッシュ | メモリ・ネットワーク負荷が軽い、UX最高 |
| 500~1,000件 | △ パフォーマンス計測して判断 | 初回ロード時間を計測し、3秒以上なら移行検討 |
| 1,000件以上 | ❌ サーバーサイドページネーション | メモリ・初回ロード時間が問題になる |

### 9.3 本プロジェクトの各API

**現状の想定件数（1契約あたり）:**

| API | 現状の想定件数 | キャッシュ利用 |
|-----|--------------|--------------|
| グループ一覧 | ~20件 | ✅ 利用する |
| ユーザー一覧 | ~100件 | ✅ 利用する |
| 契約一覧 | ~100件 | ✅ 利用する |
| お知らせ一覧 | ~20件 | ✅ 利用する |
| ドキュメント一覧 | ~100件 | ✅ 利用する |

**注:** 将来的にデータ件数が増加した場合は、個別に戦略を見直します。

### 9.4 キャッシュ仕様

#### キャッシュ有効期間

| 項目 | 推奨値 |
|------|--------|
| staleTime（新鮮さ期間） | 5分 |
| cacheTime（メモリ保持期間） | 30分 |

**注:** 上記は推奨値であり、データの更新頻度に応じて調整してください。

#### キャッシュ無効化タイミング

| タイミング | APIコール | 説明 |
|-----------|----------|------|
| 初回ページ表示 | ✅ する | キャッシュがない場合 |
| ページ再表示（キャッシュ有効） | ❌ しない | キャッシュを使用 |
| ページ再表示（キャッシュ期限切れ） | ✅ する | 最新データを取得 |
| 検索・フィルタリング | ❌ しない | クライアント側で処理 |
| ソート | ❌ しない | クライアント側で処理 |
| ページング | ❌ しない | クライアント側で処理 |
| **自身のCRUD操作後** | ✅ する | キャッシュ無効化して再取得 |
| **他ユーザーのCRUD操作** | ❌ しない | キャッシュ期限切れまで反映されない |
| 手動リフレッシュ | ✅ する | 最新データを取得 |

### 9.5 他ユーザーの操作への対応

**重要:** クライアント側キャッシュは、他のユーザーが行ったCRUD操作を自動的には反映しません。

#### 対応方法

以下のいずれかの方法で対応してください：

1. **手動リフレッシュボタン**（推奨）: UIに「更新」ボタンを配置
2. **定期ポーリング**（オプション）: データの鮮度が重要な場合、定期的にキャッシュを無効化
3. **WebSocket/Server-Sent Events**（将来的な拡張）: リアルタイム性が重要な場合

**実装例は[API実装ベストプラクティス](./implementation-best-practices.md#クライアント側キャッシュ実装)を参照してください。**

### 9.6 メモリ使用量の目安

| データ件数 | 1件あたりサイズ | 合計サイズ | 評価 |
|-----------|---------------|-----------|------|
| 100件 | 500 bytes | 50 KB | ✅ 問題なし |
| 200件 | 500 bytes | 100 KB | ✅ 問題なし |
| 1000件 | 500 bytes | 500 KB | △ 許容範囲 |
| 5000件 | 500 bytes | 2.5 MB | ❌ キャッシュ非推奨 |

### 9.7 推奨する実装パターン

1. ✅ **全件取得**: `GET /groups?per_page=1000`
2. ✅ **クライアント側処理**: 検索・ソート・ページング
3. ✅ **キャッシュ有効期間**: 5分
4. ✅ **CRUD後の無効化**: 自身の操作後は即座に反映
5. ✅ **手動リフレッシュ**: ユーザーが明示的に最新データ取得可能

**実装詳細は[API実装ベストプラクティス](./implementation-best-practices.md)を参照してください。**

### 9.8 注意事項

- **他ユーザーの操作は即座に反映されない**: キャッシュ期限切れまで、または手動リフレッシュまで反映されません
- **データの鮮度要件を確認**: リアルタイム性が必要な場合はキャッシュ期間を短くするか、ポーリング/WebSocketを検討
- **メモリ使用量に注意**: 大量データのキャッシュはクライアント側のメモリを圧迫します
- **パフォーマンス問題が発生した場合**: サーバーサイドページネーションへの移行を検討（移行は比較的容易）

### 9.9 サーバーサイドページネーションへの移行

将来的にデータ件数が増加し、パフォーマンス問題が発生した場合は、サーバーサイドページネーションへの移行を検討します。

#### 移行判断基準

| 指標 | 閾値 | 対応 |
|------|------|------|
| データ件数 | 1,000件以上 | 移行を検討 |
| 初回ロード時間 | 3秒以上 | 移行を推奨 |
| メモリ使用量 | 2 MB以上 | 移行を推奨 |
| ユーザーフィードバック | 「重い」との報告 | 移行を検討 |

#### サーバーサイドページネーション仕様

| 項目 | 仕様 |
|------|------|
| キャッシュ | 全件キャッシュしない（短時間キャッシュは許容） |
| ページネーション | サーバーサイドで実施 |
| 検索・フィルタリング | サーバーサイドで実施 |
| ソート | サーバーサイドで実施 |
| デバウンス | 検索入力時は必須（500ms推奨） |

**実装詳細は[API実装ベストプラクティス - サーバーサイドページネーション](./implementation-best-practices.md#サーバーサイドページネーション実装)を参照してください。**

## 10. ページネーション

### 10.1 本プロジェクトの方針

**本プロジェクトでは[クライアントサイドキャッシュ戦略](#9-クライアントサイドキャッシュ戦略)を採用しています。**

一覧APIでは、初回に全件データを取得してクライアント側でキャッシュし、ページネーション処理はクライアント側（JavaScript/TypeScript）で行います。これにより、ページ切り替え時のAPIコール数を削減し、高速なユーザー体験を実現します。

### 10.2 クライアント側ページネーション（本プロジェクトで採用）

クライアント側でキャッシュされたデータに対して、`Array.slice()`を使用してページング処理を行います。

**実装方法・コード例については、[API実装ベストプラクティス - クライアント側ページネーション実装](./implementation-best-practices.md#クライアント側ページネーション実装)を参照してください。**

#### 処理の流れ

1. **初回**: 全件データを取得（`GET /users?per_page=1000`）してキャッシュ
2. **ページ切り替え**: キャッシュされたデータを`Array.slice()`でページング
3. **ページネーション情報**: `total`, `totalPages`などをクライアント側で計算

#### 利点

- ✅ ページ切り替え時にAPIコール不要（即座に表示）
- ✅ サーバー負荷の軽減
- ✅ ネットワーク遅延なし
- ✅ オフライン対応可能

#### 注意事項

- 初回ロード時に全件取得するため、データ件数が多い場合は初回ロード時間が長くなる
- データ件数が1,000件を超える場合は、サーバーサイドページネーションへの移行を検討

---

### 10.3 サーバーサイドページネーション（データ件数が多い場合）

> **注:** 将来的にデータ件数が増加し、パフォーマンス問題が発生した場合に使用します。現時点では使用しません。

#### クエリパラメータ

| パラメータ | 型 | デフォルト | 説明 |
|-----------|-----|----------|------|
| page | integer | 1 | ページ番号（1以上） |
| per_page | integer | 10 | 1ページあたりの件数（1〜1000） |

#### per_page の仕様

| 項目 | 値 |
|------|-----|
| 最小値 | 1 |
| 最大値 | 1000 |
| デフォルト値 | 10 |

**注:**
- `per_page`には1〜1000の範囲で任意の整数を指定できます
- 範囲外の値が指定された場合はバリデーションエラーを返します

**使用例:**
```
GET /groups?page=1&per_page=20       # 1ページ目（1-20件目）
GET /groups?page=2&per_page=20       # 2ページ目（21-40件目）
GET /groups?page=1&per_page=100      # 100件ずつ取得
```

#### レスポンス形式

レスポンスには以下のページネーション情報が含まれます：

```json
{
  "success": true,
  "data": {
    "total": 100,        // 検索条件に一致した総件数
    "page": 1,           // 現在のページ番号
    "per_page": 20,      // 1ページあたりの件数
    "totalPages": 5,     // 総ページ数（total ÷ per_page の切り上げ）
    "items": [...]       // データ配列（キー名はエンドポイントにより異なる）
  },
  "timestamp": "2025-11-01T10:00:00+09:00"
}
```

**注:** データ配列のキー名は、エンドポイントにより `users`, `groups`, `contracts` など異なります。

#### バリデーションエラー時のレスポンス

無効な`per_page`値が指定された場合：

**範囲外の値（1000超過、または0以下）:**

```json
{
  "success": false,
  "error": {
    "code": "VALIDATION_ERROR",
    "message": "ページネーションパラメータが不正です",
    "details": {
      "per_page": ["1ページあたりの件数は1〜1000の範囲で指定してください"]
    }
  },
  "timestamp": "2025-11-01T10:00:00+09:00"
}
```

**HTTPステータス:** 422 Unprocessable Entity

## 11. ソート

### 11.1 本プロジェクトの方針

**本プロジェクトでは[クライアントサイドキャッシュ戦略](#9-クライアントサイドキャッシュ戦略)を採用しています。**

一覧APIでは、初回に全件データを取得してクライアント側でキャッシュし、ソート処理はクライアント側（JavaScript/TypeScript）で行います。これにより、ソート順変更時のAPIコール数を削減し、即座にソート結果を表示できます。

### 11.2 クライアント側ソート（本プロジェクトで採用）

クライアント側でキャッシュされたデータに対して、`Array.sort()`を使用してソート処理を行います。

**実装方法・コード例については、[API実装ベストプラクティス - クライアント側ソート実装](./implementation-best-practices.md#クライアント側ソート実装)を参照してください。**

#### 処理の流れ

1. **キャッシュされたデータを取得**: React Queryなどから全件データを取得
2. **配列をコピー**: `[...allUsers]` で元の配列を変更しないようコピー
3. **ソート処理**: `Array.sort()`でソート（文字列は`localeCompare()`、日付は`Date`オブジェクトで比較）
4. **ソート順を適用**: `asc` または `desc` に応じて比較結果を反転

#### ソート可能なフィールド（例）

各APIエンドポイントでソート可能なフィールドは異なります。以下は一般的な例です：

| フィールド | 型 | 説明 | ソート方法 |
|-----------|-----|------|----------|
| name | string | 名前 | `localeCompare()` 使用 |
| email | string | メールアドレス | `localeCompare()` 使用 |
| createdAt | string | 作成日時 | `Date` オブジェクトで比較 |
| updatedAt | string | 更新日時 | `Date` オブジェクトで比較 |
| status | string | ステータス | 文字列比較 |

**注:** 各APIエンドポイントの詳細は、個別のAPI仕様書を参照してください。

#### 利点

- ✅ ソート順変更時にAPIコール不要（即座に表示）
- ✅ サーバー負荷の軽減
- ✅ ネットワーク遅延なし
- ✅ 複数フィールドでの複合ソートも容易に実装可能

#### 注意事項

- `Array.sort()` は元の配列を変更するため、必ずコピーしてからソート処理を行う
- 日本語のソートは `localeCompare('ja')` を使用することで自然な順序になる
- データ件数が1,000件を超える場合は、サーバーサイドソートへの移行を検討

---

### 11.3 サーバーサイドソート（データ件数が多い場合）

> **注:** 将来的にデータ件数が増加し、パフォーマンス問題が発生した場合に使用します。現時点では使用しません。

#### クエリパラメータ

## 15. トランザクション管理

すべてのAPIエンドポイントで共通するデータベーストランザクション管理の方針を定義します。

### 15.1 デフォルト設定

| 項目 | 設定値 | 理由 |
|------|--------|------|
| **分離レベル** | READ COMMITTED | デフォルト設定。ダーティリードを防ぎつつ、パフォーマンスと整合性のバランスを確保 |
| **ロック戦略** | 行レベルロック | 同時実行性を最大化。テーブルロックは使用しない |
| **タイムアウト** | 30秒 | デッドロック防止。長時間ロックを防ぐ |
| **自動コミット** | 無効 | 明示的なトランザクション制御を行う |

### 15.2 トランザクション分離レベル

| 分離レベル | 使用場面 | 特徴 |
|-----------|---------|------|
| **READ COMMITTED** | ほとんどのAPI（デフォルト） | ダーティリード防止、ファントムリード許容 |
| **REPEATABLE READ** | 一貫性が重要な処理 | 同一トランザクション内で同じデータを読む場合 |
| **SERIALIZABLE** | 厳密な整合性が必要な処理 | パフォーマンス低下のため慎重に使用 |

**注:** デフォルト以外の分離レベルを使用する場合は、個別のAPI処理設計書で明記してください。

### 15.3 トランザクション開始・終了タイミング

#### トランザクション開始

以下の条件を**すべて満たす**場合にトランザクションを開始します：

1. データベースへの書き込み操作（INSERT/UPDATE/DELETE）がある
2. バリデーションが完了している
3. 認証・認可チェックが完了している

**トランザクションを開始しない場合:**
- 読み取り専用（SELECT）のみのAPI
- バリデーションエラーの可能性がある処理の前
- 認証・認可チェックの前

**理由:** バリデーションエラーや認証エラーでトランザクションを開始すると、無駄なリソース消費とロールバック処理が発生するため。

#### トランザクション終了（コミット）

以下の条件を**すべて満たす**場合にコミットします：

- すべてのデータベース操作が正常に完了
- 外部API連携がある場合は、そのレスポンスも正常
- ビジネスロジックエラーが発生していない
- タイムアウトが発生していない

#### トランザクション終了（ロールバック）

以下のいずれかのエラーが発生した場合、トランザクションをロールバックします：

| エラー種別 | 発生タイミング | ロールバック実施 | 理由 |
|-----------|--------------|----------------|------|
| バリデーションエラー | トランザクション開始前 | ❌ 不要 | トランザクション未開始 |
| 認証・認可エラー | トランザクション開始前 | ❌ 不要 | トランザクション未開始 |
| リソース不在エラー | 読み取り処理中 | ❌ 不要 | データ変更なし |
| **競合エラー** | 更新処理中 | ✅ **実施** | データ整合性保持 |
| **データベースエラー** | SQL実行時 | ✅ **実施** | データ整合性保持 |
| **外部API連携エラー** | API呼び出し時（トランザクション開始後） | ✅ **実施** | データ整合性保持 |
| **ビジネスロジックエラー** | ビジネスルール違反検出時 | ✅ **実施** | データ整合性保持 |
| **タイムアウト** | 処理時間超過 | ✅ **実施** | 不完全なデータ登録防止 |

**注:** エラー分類の詳細は[エラーハンドリング方針](#7-エラーハンドリング方針)セクションを参照してください。

#### ロールバック失敗時の対応

ロールバック処理自体が失敗した場合（稀なケース）：

1. **エラーログに詳細を記録**
   - リクエストID
   - トランザクションID
   - 失敗理由
   - 実行されたSQL履歴
   - データベース接続状態

2. **システム管理者へアラート通知**
   - 即座に通知（メール・Slack等）
   - 緊急度: Critical

3. **HTTPステータス 500 でエラーレスポンス返却**
   - クライアントには「サーバーエラー」として返却
   - リクエストIDを含める（手動調査用）

4. **手動確認の準備**
   - トランザクションIDをログに記録
   - データベースの状態を確認できるよう、関連テーブルのスナップショットを保存（可能な場合）

### 15.4 ロック戦略

#### 基本方針

- **行レベルロック**: 標準使用
- **テーブルロック**: 原則使用禁止（メンテナンス時のみ）
- **楽観的ロック**: バージョン管理カラム（version）を使用する場合
- **悲観的ロック**: SELECT ... FOR UPDATE を使用する場合

#### 楽観的ロック vs 悲観的ロック

| 項目 | 楽観的ロック | 悲観的ロック |
|------|------------|------------|
| **使用場面** | 競合が少ない処理 | 競合が多い処理、在庫管理など |
| **実装方法** | versionカラムで更新時チェック | SELECT ... FOR UPDATE |
| **競合時** | 更新失敗、リトライまたはエラー | ロック待ち（タイムアウトあり） |
| **パフォーマンス** | 高い | 低い（ロック待ち発生） |

**記入例（楽観的ロック）:**
```sql
-- 読み取り時
SELECT id, name, version FROM users WHERE id = :userId;

-- 更新時
UPDATE users
SET name = :name, version = version + 1
WHERE id = :userId AND version = :currentVersion;

-- 影響行数が0の場合は競合エラー（409 Conflict）
```

**記入例（悲観的ロック）:**
```sql
-- ロック取得
SELECT id, stock FROM products WHERE id = :productId FOR UPDATE;

-- 在庫更新
UPDATE products SET stock = stock - :quantity WHERE id = :productId;
```

### 15.5 デッドロック対策

**デッドロック防止策:**

1. **ロック順序の統一**: 複数テーブルをロックする場合、常に同じ順序でアクセス
2. **ロック時間の最小化**: トランザクション内の処理を最小限にする
3. **タイムアウト設定**: 30秒でタイムアウト
4. **インデックス利用**: WHERE句で使用するカラムにインデックスを設定し、ロック範囲を最小化

**デッドロック発生時の対応:**

1. エラーログに記録（関連するSQL、テーブル、トランザクションID）
2. トランザクションをロールバック
3. HTTPステータス 500 でエラーレスポンス返却
4. リトライは行わない（クライアント側で判断）

### 15.6 トランザクション管理のベストプラクティス

1. ✅ **トランザクションは短く保つ**: 不要な処理はトランザクション外で実施
2. ✅ **外部API呼び出しはトランザクション外**: ネットワーク遅延でロック時間が延びる
3. ✅ **ファイルI/Oはトランザクション外**: I/O待ちでロック時間が延びる
4. ✅ **バリデーションはトランザクション前**: 入力チェックでトランザクション時間を無駄にしない
5. ❌ **トランザクション内でのループ処理は避ける**: バッチ処理を検討

## 16. ログ出力ポリシー

すべてのAPIエンドポイントで共通するログ出力の方針を定義します。

### 16.1 ログレベル定義

| ログレベル | 使用場面 | 出力内容 |
|-----------|---------|---------|
| **ERROR** | エラー発生時 | HTTPステータス 500系、データベースエラー、外部API連携エラー |
| **WARN** | 警告事象発生時 | HTTPステータス 400系、バリデーションエラー、認証・認可エラー |
| **INFO** | 通常処理 | API呼び出し開始・終了、主要な処理ステップ |
| **DEBUG** | 開発・調査用 | 詳細な処理内容、変数値（本番環境では無効化） |

### 16.2 必須ログ出力項目

すべてのAPIで以下の項目をログ出力します：

#### リクエスト開始時（INFO）

```json
{
  "level": "INFO",
  "timestamp": "2025-11-01T10:00:00.123+09:00",
  "requestId": "req_abc123",
  "method": "POST",
  "endpoint": "/api/v1/users",
  "userId": "usr_001",
  "contractId": "cnt_001",
  "ipAddress": "192.168.1.100",
  "userAgent": "Mozilla/5.0..."
}
```

#### レスポンス返却時（INFO）

```json
{
  "level": "INFO",
  "timestamp": "2025-11-01T10:00:00.456+09:00",
  "requestId": "req_abc123",
  "method": "POST",
  "endpoint": "/api/v1/users",
  "httpStatus": 200,
  "processingTime": 123,
  "userId": "usr_001",
  "contractId": "cnt_001"
}
```

**注:** `processingTime` はミリ秒単位

#### エラー発生時（ERROR/WARN）

```json
{
  "level": "ERROR",
  "timestamp": "2025-11-01T10:00:00.789+09:00",
  "requestId": "req_abc123",
  "method": "POST",
  "endpoint": "/api/v1/users",
  "httpStatus": 500,
  "errorCode": "INTERNAL_ERROR",
  "errorMessage": "データベース接続に失敗しました",
  "userId": "usr_001",
  "contractId": "cnt_001",
  "stackTrace": "..."
}
```

### 16.3 出力禁止項目（機密情報）

以下の情報は**絶対にログ出力してはいけません**：

#### 認証情報
- ❌ パスワード（平文・ハッシュ化後問わず）
- ❌ 認証トークン（JWT、アクセストークン、リフレッシュトークン）
- ❌ APIキー
- ❌ セッションID（トレーシング以外の目的）

#### 個人情報
- ❌ クレジットカード番号
- ❌ マイナンバー
- ❌ 生年月日（フルフォーマット）
- ❌ メールアドレス（一部マスキングなしの場合）
- ❌ 電話番号（一部マスキングなしの場合）
- ❌ 住所（詳細）

#### その他機密情報
- ❌ 暗号化キー
- ❌ Salt値
- ❌ データベース接続文字列（パスワード部分）

### 16.4 マスキング方針

個人情報を出力する必要がある場合は、以下のルールでマスキングします：

| 項目 | マスキング方法 | 例 |
|------|--------------|-----|
| メールアドレス | ローカル部の一部をマスク | `ya****@example.com` |
| 電話番号 | 中間4桁をマスク | `090-****-5678` |
| ユーザー名 | 姓のみ表示 | `山田****` |
| クレジットカード | 下4桁のみ表示 | `****-****-****-1234` |

### 16.5 ログローテーション

| 項目 | 設定値 |
|------|--------|
| ローテーション周期 | 日次（毎日0時） |
| 保存期間 | 90日間 |
| 圧縮 | あり（gzip） |
| ファイルサイズ上限 | 100MB（超過時は即座にローテーション） |

### 16.6 ログ保存先

| 環境 | 保存先 | 備考 |
|------|--------|------|
| 開発環境 | ローカルファイル | `/var/log/app/` |
| 検証環境 | Azure Log Analytics | クエリ・分析可能 |
| 本番環境 | Azure Log Analytics | クエリ・分析可能、アラート設定済み |
