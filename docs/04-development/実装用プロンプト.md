# 実装プロンプト

---

## 前提

- 実装方式: TDD（テストコードは作成済み。テストを通過させる形で実装を進める）
- ファイル出力先: `src/` 配下
- タスク管理: `docs/04-development/<機能名>/task_list.md` に作成・進捗管理

## 入力ファイル（実装前に全て読み込むこと）

### 共通設計書
- `docs/03-features/common/common-specification.md`
- `docs/03-features/common/ui-common-specification.md`
- `docs/03-features/common/app-database-specification.md`
- `docs/03-features/common/logging-specification.md`

### 機能別設計書
- `docs/03-features/<機能名>` 配下の全ファイル

### 既存テストコード（実装の仕様として扱う）
- `tests/unit/<機能名>/` 配下の全テストファイル
- `tests/unit/conftest.py`

### 既存実装（参照用）
- `src/config.py`
- `src/__init__.py`（アプリファクトリ）

## 実装手順

### Step 1: 仕様把握

上記の全設計書・テストコードを読み込み、実装すべき内容を把握する。
不明点・設計書とテストコードに矛盾がある場合は実装前にユーザーに報告・確認する。

### Step 2: タスク分解と task_list.md 出力

実装を以下の3層で分解し、`docs/04-development/<機能名>/task_list.md` を作成する。
機能によって不要な層はスキップしてよい。

| 層                     | 対象                                | 例                                         |
| ---------------------- | ----------------------------------- | ------------------------------------------ |
| **データ定義層**       | models, exceptions, 定数定義        | DB定義・例外クラス等、他から依存される基盤 |
| **ビジネスロジック層** | services, utility, connector        | CRUD処理・外部API連携等のコアロジック      |
| **インターフェース層** | views, forms, templates, middleware | ルーティング・バリデーション・画面描画     |

**task_list.md の形式:**

```markdown
# <機能名> 実装タスクリスト

## タスク一覧

| #   | タスク名           | 対象ファイル                     | 対応テスト               | 実装フロー状態 | 備考 |
| --- | ------------------ | -------------------------------- | ------------------------ | -------------- | ---- |
| 1   | データ定義層       | src/.../models.py, exceptions.py | tests/unit/.../test_*.py | 未着手         |      |
| 2   | ビジネスロジック層 | src/.../services.py              | tests/unit/.../test_*.py | 未着手         |      |
| 3   | インターフェース層 | src/.../views.py, forms.py       | tests/unit/.../test_*.py | 未着手         |      |

## 実装フロー状態

| 状態     | 意味                 |
| -------- | -------------------- |
| 未着手   | 実装未開始           |
| 実装中   | ① 実装 実施中        |
| テスト中 | ② pytest 実行中      |
| 修正中   | ③ テスト失敗・修正中 |
| 完了     | ④ 全テスト通過       |

## セルフレビュー進捗

| 観点                         | 状態   | 確認結果 |
| ---------------------------- | ------ | -------- |
| 機能: 設計書との差分         | 未実施 |          |
| 機能: テスト仕様カバレッジ   | 未実施 |          |
| 機能: インターフェース整合性 | 未実施 |          |
| 非機能: セキュリティ         | 未実施 |          |
| 非機能: ログ準拠             | 未実施 |          |
| 非機能: エラーハンドリング   | 未実施 |          |
```

task_list.md の内容はユーザー確認後に着手する。

### Step 3: タスクごとの実装サイクル（全タスクに繰り返す）

各タスクで以下のサイクルを実施する：

① **実装**（task_list.md の 実装フロー状態 → 「実装中」に更新）
- 設計書・テストコードの仕様を満たす最小限の実装を行う
- 過剰実装禁止（テストが要求する以上のロジックを追加しない）

② **テスト実行**（task_list.md の 実装フロー状態 → 「テスト中」に更新）
```bash
pytest tests/unit/<機能名>/<対応テストファイル> -v
```

③ **失敗時は実装修正**（task_list.md の 実装フロー状態 → 「修正中」に更新）
- エラーメッセージを分析し、原因を特定してから修正する
- 修正方針をユーザーに報告してから実施する
- テストコード自体は修正しない（設計書と矛盾がある場合はユーザーに報告）
- 修正後は ② に戻る

④ **全テスト通過を確認したらタスク完了**（task_list.md の 実装フロー状態 → 「完了」に更新）
- 次のタスクに進む

### Step 4: 全タスク完了後のセルフレビュー

以下の観点でセルフレビューを実施し、各観点の確認が完了したら task_list.md の「セルフレビュー進捗」を更新する。
全観点の確認完了後、結果をユーザーに報告する。

**機能的観点**
- 設計書の仕様と実装コードに差分がないか
- テストコードが要求する全仕様が実装されているか
- モジュール間のインターフェース（引数・戻り値・例外）が設計書と一致しているか

**非機能的観点**
- セキュリティ: インジェクション対策（ORM使用・入力値サニタイズ）、機密情報のログ出力禁止、権限チェック漏れがないか
- ログ: logging-specification.md のフィールド要件・レベル基準に準拠しているか
- エラーハンドリング: 例外が適切に捕捉・変換されているか、上位へ素通しすべき例外が握りつぶされていないか

## 制約

- `src/` 配下に存在しない新規ファイルを作成する場合は事前にユーザー確認
- テストコードは原則修正しない（テストコードが正）、修正が必要と思しき事項が発生した場合はユーザー確認
- `__init__.py` を含む既存ファイルを変更する場合は影響範囲を報告してから実施
- インポートパスは `src.` プレフィックスなし（例: `from auth.factory import ...`）
