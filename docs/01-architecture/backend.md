# バックエンドアーキテクチャ

## 概要

Databricks IoTシステムのバックエンドは、**Azure環境内に統合されたDatabricks Platform**のレイクハウスアーキテクチャを中心に、Flask Webアプリケーションとメダリオンアーキテクチャを組み合わせた構成を採用しています。Azure IoT Services（IoT Hubs、Event Hubs）とDatabricks Platformが同一Azure環境内で連携し、IoTデバイスからの大量データをリアルタイムで処理し、効率的な分析と可視化を実現します。

## 技術スタック

- **プラットフォーム**: Databricks (Premium ライセンス)
- **Webフレームワーク**: Flask (Python)
- **実行環境**: Azure App Servic
- **データ処理**: Lakeflow 宣言型パイプライン (LDP) - Python/SQL
- **データベース**:
  - Databricks Unity Catalog (Delta Lake) - レイクハウス
  - MySQL互換データベース (SingleStore互換) - OLTP
- **データ取込み**: Azure IoT Hubs → Event Hubs (Kafka Endpoint) → Event Hubs Capture (ADLS)
- **通信プロトコル**: MQTT (IoTデバイス)
- **認証**: 共通認証モジュール（Azure/AWS/オンプレミス対応）。OAuth トークン フェデレーションによるDatabricks接続
- **ネットワーク**: Private Link (Event Hub/ADLS/MySQL/Databricks UI API)
- **Databricks接続**: @databricks/sql (SQL Warehouse)

## 主要な設計原則

1. **メダリオンアーキテクチャ**
   - ブロンズ層: 生データの保存とアラート判定
   - シルバー層: データの構造化・変換・異常検知
   - ゴールド層: BI・分析用の最適化されたデータ
   - 段階的なデータ品質向上

2. **データストアの使い分け**
   - レイクハウス (Databricks): センサーデータ、分析用途
   - OLTP (MySQL互換): マスタデータ、デバイスステータス
   - 適材適所のデータ配置によるパフォーマンス最適化

3. **リアルタイム処理**
   - ストリーミングデータパイプライン (Kafka → LDP)
   - リアルタイムアラート・異常検知
   - スケーラブルなデータ処理基盤

4. **拡張可能性**
   - 新しいIoTデバイスへの容易な対応
   - ライブラリによる機能拡張
   - モジュラーな設計

## レイヤー詳細

### 1. IoT Device Layer
- MQTTプロトコルによるデータ送信
- 各種IoTセンサー・デバイスからのデータ収集
- デバイスへの下り電文送信 (IoT Hubs経由)
  - 今回のスコープ対象外とする

### 2. Data Ingestion Layer
- **Azure IoT Hubs**: IoTデバイスとの通信ハブ
- **Event Hubs**: Kafkaエンドポイント提供
  - **Event Hubs Capture**: RAWデータをADLSに継続出力（長期保存・障害復旧用）
- MQTT → REST変換 (必要に応じて)
- ストリーミングデータの受信
- **Private Link**: Event HubsへのPrivate Link接続

### 3. Databricks Processing Layer
- **Lakeflow 宣言型パイプライン (LDP)**: Python/SQLでの実装
- **ブロンズ層**: Event Hubs Captureによる生データ保存（LDP対象外）
- **シルバー層**: 電文解析・構造化、異常データ検出、OLTP更新
- **ゴールド層**: BI・分析用に最適化されたデータ
- アラート処理の実装
- **コンピューティングプレーン**:
  - サーバレス: LDPパイプライン（Databricksマネージド）
  - クラシック (VMSS): Compute Cluster（顧客VNet内デプロイ）
- **Private Link**: ADLS/MySQL/Databricks UI APIへのPrivate Link接続

### 4. Application Layer (Flask)
- **管理画面**: マスタ管理、設定変更、CRUD操作
- **ユーザー識別**: リバースプロキシヘッダからのユーザー情報取得
  - **認証**: 共通認証モジュール（Azure/AWS/オンプレミス対応）。OAuth トークン フェデレーションによるDatabricks接続
  - アプリではリバースプロキシが付与するヘッダ情報からユーザーを識別
- **データアクセス**: SQL WarehouseとOLTPデータベースの使い分け

### 5. Data Layer
- **Databricks Unity Catalog**:
  - Delta Lakeによるデータストレージ
  - **ADLS (Azure Data Lake Storage)**: ストレージバックエンド
  - センサーデータの大量保存
  - SQL Warehouse経由でのアクセス
  - **Private Link**: ADLS用Private Link (dfs)経由でのデータ読み書き
- **MySQL互換データベース**:
  - マスタデータ管理
  - デバイスステータスの保存
  - SingleStore互換
  - **Private Link**: MySQL用Private Link経由でのアクセス
  - **踏み台VM**: プライベートネットワーク内のDB保守アクセス

## セキュリティ対策

- **認証基盤**:
  - **認証モジュール**: 共通モジュール化により、「Azure環境(Easy Auth)」、「AWS環境(ALB＋cognito)」、「オンプレミス環境(自前認証)」の三パターンに対応可能
  - **Databricks接続**: OAuth トークン フェデレーションを用いてユーザー単位の認証を行い、データスコープ制御を実現
  - **アクセス元IP制限**: Databricksワークスペース、公開フロントへのIP制限
- **リバースプロキシ**: 認証成功後、ユーザ識別子・アクセストークンをリクエストヘッダに付与
- **Flaskアプリ**: リバースプロキシヘッダからユーザー情報を取得してユーザー識別
- **ユーザ権限管理**: サービスプリンシパル権限とユーザ認可APIの使い分け
- **データアクセス制御**: 動的ビューによる組織階層ベースのデータ範囲制限
  - `organization_closure` テーブルを使用した下位組織の取得
  - ロールは動線制御（ボタン、メニューの表示/非表示）のみに使用
- **Private Linkによるセキュア通信**:
  - Event Hub用Private Link（ストリーミング処理）
  - ADLS用Private Link (dfs)（データ読み書き）
  - MySQL用Private Link（OLTP DBアクセス）
  - Databricks UI/API用Private Link（クラシックコンピューティングプレーン）
- **Azure DNS**: Private Link名前解決のためのPrivate Zones
- **外部通信制限**: セキュリティポリシーによる外部向け通信の制御
- **環境変数管理**: 接続情報・機密情報の環境変数化
- **監査ログ**: Databricksシステムテーブルへの出力

## パフォーマンス最適化

- **メダリオンアーキテクチャ**: 段階的なデータ処理による効率化
- **データストアの使い分け**:
  - レイクハウス: センサーデータ
  - OLTP: マスタ・デバイスステータス
- **ストリーミング処理**: リアルタイムでのデータ取り込みと処理
- **Databricks Compute**: スケーラブルな処理基盤
- **データ保持期間の最適化**: ブロンズ層の短期保持による容量削減
- **効率的なクエリ**: SQL Warehouse経由の最適化されたアクセス

## 拡張性と保守性

- **モジュラー設計**: Flask App、LDP、データ層の明確な分離
- **新規デバイス対応**: LDPの拡張による柔軟なデバイス追加
- **ライブラリ拡張**: Pythonライブラリによる機能追加の容易性
- **環境分離**: ローカル開発・検証・本番の3環境の明確な分離
- **デプロイ管理**:
  - App Compute: GitリポジトリからのPull、またはアセット圧縮ファイルのアップロード
  - Databricks リソース: Databricks CLIによる管理
  - Databricks アセット バンドルの活用
- **ログ管理**: Unity Catalogまたは外部ログアダプターへの出力
- **監視**: ヘルスチェックエンドポイント (App Compute URL)

## 今後の改善点

- **App Computeのシングルトン保証**: コンテナの重複実行・中断の影響調査

- **ログの永続化**: App Computeコンテナ再起動時のログ消失対策
