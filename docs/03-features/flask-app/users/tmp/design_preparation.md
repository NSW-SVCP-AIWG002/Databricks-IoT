### 1.2 アクセス権限（FR-004-9）

| 機能 | システム保守者 | 管理者 | 販社ユーザ | サービス利用者 |
|------|---------------|--------|-----------|---------------|

### 1.3 非機能要件

#### NFR-SEC-001: 認証・認可
- **認証方式**: Databricks User認証（Databricks標準機能、Flaskアプリでは実装しない）
- **ユーザー識別**: リバースプロキシヘッダからユーザー情報を取得
- **4つのロール**: システム保守者、管理者、販社ユーザ、サービス利用者

#### NFR-SEC-006: ロールベースアクセス制御要件
- Flask routeにデコレーター実装 (`@role_required('管理者', 'システム保守者')`)
- 権限不足時は403 Forbiddenエラーと分かりやすいメッセージを表示

#### NFR-SEC-007: データスコープ制限要件
- **販社ユーザ**: 自社に紐づくユーザーのみアクセス可能
  - SQL WHERE句: `WHERE sales_company_id = :user_sales_company_id`
- **サービス利用者**: 自社データのみアクセス可能
  - SQL WHERE句: `WHERE user.user_id = :current_user_id`
- **OLTP DB**: WHERE句によるフィルタリング実装

#### NFR-PERF-003: Web画面応答性能
- 初期ページ読み込み時間: 3秒以内
- 画面遷移時間: 1秒以内
- API応答時間: 500ms以内

## 2. 解析した不足情報（TODO項目）

### 2.1 検索条件項目の未定義（FR-004-2-1）

**現状**: 機能要件書に「検索条件項目: TODO」と記載されており、具体的な検索項目が定義されていない

**推奨される検索条件**:
以下の検索条件を設計時に追加することを推奨します：

1. **キーワード検索**
   - 対象: ユーザ名、メールアドレス
   - 検索方式: 部分一致（LIKE検索）

2. **ユーザ種別（権限）フィルタ**
   - 選択肢: システム保守者、管理者、販社ユーザ、サービス利用者
   - 選択方式: 複数選択可能

3. **所属（組織）フィルタ**
   - 選択肢: 組織一覧から選択
   - 選択方式: 単一選択または複数選択
   - データスコープ制限: 販社ユーザの場合は自社組織のみ表示

4. **地域フィルタ**（オプション）
   - 選択肢: 地域マスタから選択

5. **ステータスフィルタ**（オプション）
   - 選択肢: 有効、無効

**理由**: ユーザ一覧画面では、大量のユーザデータから目的のユーザを素早く検索するために、複数の検索条件が必要です。

### 2.2 データベーステーブル定義の未確認

**現状**: 機能要件書にはユーザ管理機能の表示項目が記載されているが、データベーステーブル定義（カラム名、データ型、制約等）が明示されていない

7. インデックス設計（特にデータスコープ制限用のorganization_id、sales_company_id、customer_id）

**対応方針**: データベース設計書（`01-architecture/database.md`）を参照して、usersテーブルの物理設計を確認し、UI仕様書とワークフロー仕様書に反映する必要があります。

### 2.3 Databricks連携仕様の詳細

**現状**: 機能要件書には「Databricks Userへの登録連携」「Databricksワークスペースグループへのユーザーの登録連携」と記載されているが、具体的な連携仕様が不明

**必要な確認事項**:
1. Databricks SCIM APIのエンドポイント
2. 認証方式（サービスプリンシパルのトークン使用）
3. ユーザ作成時のリクエストボディ構造
4. グループ紐付け時のリクエストボディ構造
5. エラーハンドリング（Databricks側のエラー時の対応）

### 2.4 メールアドレスの重複チェック

**現状**: 機能要件書には明示されていないが、一般的にメールアドレスはユーザの一意識別子として使用されるため、重複チェックが必要

**推奨される実装**:
1. サーバーサイドバリデーション: メールアドレスの重複チェック（UNIQUE制約）
2. エラーメッセージ: 「このメールアドレスは既に使用されています」

### 2.5 バリデーションルールの詳細

**現状**: 機能要件書には「バリデーション（必須チェック、形式チェック、重複チェック）」と記載されているが、各項目の具体的なバリデーションルールが不明

**推奨されるバリデーションルール**:
1. **ユーザID**: 自動採番（編集不可）
2. **ユーザ名**: 必須、最大255文字
3. **メールアドレス**: 必須、メールアドレス形式（RFC 5322）、最大255文字、重複チェック
4. **ユーザ種別（権限）**: 必須、許可された値のみ（システム保守者、管理者、販社ユーザ、サービス利用者）
5. **地域**: 任意、最大100文字
6. **住所**: 任意、最大500文字
7. **所属（組織名）**: 必須、既存の組織IDのみ

## 3. 新設した章の詳細

### 3.1 テンプレート外の新設章: なし

**理由**: UI仕様書テンプレートとワークフロー仕様書テンプレートに記載されている標準的な章構成で、ユーザー管理機能の仕様を十分に記載できるため、テンプレート外の新設章は不要と判断しました。

### 3.2 Flask SSRルート設計の明確化

**背景**: 本システムはFlask SSR（Server-Side Rendering）を採用しているため、REST APIではなくFlaskルートとして設計する必要があります。

**Flask SSRの特徴**:
- JSONレスポンスではなく、HTMLレスポンス（`render_template()`）またはリダイレクト（`redirect()`）を返す
- ルートは `/api/users` ではなく `/admin/users` の形式
- クライアントサイドJavaScriptは最小限（Jinja2テンプレートでサーバーサイドレンダリング）

**ルート設計**:

| 操作 | HTTPメソッド | エンドポイント | レスポンス形式 | 用途 |
|------|-------------|---------------|---------------|------|
| 一覧表示 | GET | `/admin/users` | HTML | ユーザー一覧画面表示 |
| 詳細表示 | GET | `/admin/users/<user_id>` | HTML | ユーザー詳細画面表示（モーダル） |
| 登録画面 | GET | `/admin/users/create` | HTML | ユーザー登録フォーム表示（モーダル） |
| 登録実行 | POST | `/admin/users/create` | リダイレクト (302) | ユーザー登録処理、成功時一覧へリダイレクト |
| 更新画面 | GET | `/admin/users/<user_id>/edit` | HTML | ユーザー更新フォーム表示（モーダル） |
| 更新実行 | POST | `/admin/users/<user_id>/update` | リダイレクト (302) | ユーザー更新処理、成功時一覧へリダイレクト |
| 削除実行 | POST | `/admin/users/<user_id>/delete` | リダイレクト (302) | ユーザー削除処理、成功時一覧へリダイレクト |

**注**: 本システムではDELETEメソッドではなく、POSTメソッドで削除処理を実装します（HTMLフォームの制約のため）。

## 4. 設計方針

### 4.1 UI設計方針

1. **CRUD画面テンプレートの遵守**: 機能要件書に記載されている「CRUD画面テンプレート」に従い、すべての操作（一覧・登録・更新・参照・削除）で統一されたUI構成を採用します。

2. **モーダル表示**: 登録・更新・参照画面は、一覧画面内でモーダルとして表示します（別ページ遷移なし）。

3. **確認モーダル**: 登録・更新・削除処理実行時には、確認モーダルを表示してユーザーに再確認を促します。

4. **レスポンシブデザイン**: 画面サイズ1920×1080を基準としますが、タブレット（768x1024以上）でも操作可能なレスポンシブデザインを採用します。

### 4.2 ワークフロー設計方針

1. **データスコープ制限の実装**: すべてのデータ取得処理において、ログインユーザーのロール（システム保守者、管理者、販社ユーザ、サービス利用者）に応じたデータスコープ制限を実装します。

2. **Databricks連携の統合**: ユーザ登録・更新・削除時には、OLTP DBへの操作とDatabricks API呼び出しを連鎖的に実行し、エラー時には適切にロールバックします。

3. **パフォーマンス最適化**: ページング、インデックス活用、SQL最適化により、応答性能要件（初期ページ読み込み3秒以内、API応答500ms以内）を満たします。

4. **セキュリティ実装**:
   - Flask routeにデコレーター実装 (`@role_required`)
   - SQLインジェクション対策（SQLAlchemy ORM使用）
   - XSS対策（Jinja2自動エスケープ）
   - CSRF対策（Flask-WTF CSRF保護）
