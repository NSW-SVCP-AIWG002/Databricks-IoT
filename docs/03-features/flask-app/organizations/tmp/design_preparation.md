# 組織管理機能 設計準備資料

## 1. 把握した要件

### 1.1 機能概要

組織管理機能（FR-004-4）は、Databricks IoTシステムにおける組織マスタの管理を行う機能です。組織の一覧表示、検索、参照、登録、更新、削除の基本的なCRUD操作に加え、Databricks ワークスペースグループとの連携を行います。

### 1.2 機能要件（functional-requirements.md より）

#### FR-004-4-1: 組織一覧・検索
- **表示項目:**
  - 組織ID
  - 組織名
  - 組織種別
  - 住所
  - 電話番号
  - FAX
  - 担当者名
  - 契約状態
  - 契約開始日
  - 契約終了日
  - 関連組織名
- **検索条件:**
  - 組織ID（完全一致）
  - 組織名（部分一致）
  - 組織種別（選択）
  - 契約状態（選択: 契約中/契約終了）
  - 担当者名（部分一致）
  - 関連組織名（部分一致）

#### FR-004-4-2: 組織登録
- **入力項目:**
  - 組織名
  - 組織種別
  - 住所
  - 電話番号
  - FAX
  - 担当者名
  - 契約状態
  - 契約開始日
  - 契約終了日
  - 関連組織名
- **外部連携:** Databricks ワークスペースグループの作成連携
- **備考:**
  - 組織登録時にDatabricksワークスペースグループを自動作成
  - 組織IDはシステム内で自動採番

#### FR-004-4-3: 組織更新
- **入力項目:** 上記入力項目と同じ
- **外部連携:** Databricks ワークスペースグループの更新連携
- **備考:** 組織IDは更新不可

#### FR-004-4-4: 組織参照
- **表示項目:** 上記表示項目と同じ

#### FR-004-4-5: 組織削除
- **削除方式:** 論理削除
- **外部連携:** Databricks ワークスペースグループの削除連携
- **備考:** 組織削除時に配下にある組織またはユーザの削除は行わない

### 1.3 画面仕様（functional-requirements.md より）

#### 画面一覧
| 画面ID | 画面名 | スラッグ | 概要 |
|--------|--------|----------|------|
| ADM-009 | 組織一覧画面 | /admin/organizations | 組織の一覧・検索/削除 |
| ADM-010 | 組織登録画面 | /admin/organizations/register | 組織の登録 |
| ADM-011 | 組織更新画面 | /admin/organizations/update | 組織の更新 |
| ADM-012 | 組織参照画面 | /admin/organizations/detail/:id | 組織の詳細情報表示 |

#### 画面共通要件
- 画面サイズ: 1920×1080
- 画面左部にグローバルメニューカラムを表示
- 登録、更新、参照画面は同画面内でモーダルとして表示
- 登録系ボタンは青系、削除ボタンは赤系
- 削除ボタンはチェックボックス選択時のみ活性

### 1.4 アクセス権限（functional-requirements.md より）

| 機能 | システム保守者 | 管理者 | 販社ユーザ | サービス利用者 |
|------|---------------|--------|-----------|---------------|
| 組織一覧 | ◎ | ◎ | ○ | - |
| 組織参照 | ◎ | ◎ | ○ | - |
| 組織登録 | ◎ | ◎ | ○ | - |
| 組織更新 | ◎ | ◎ | ○ | - |
| 組織削除 | ◎ | ◎ | ○ | - |

**備考:**
- 販社ユーザの「○」は、自社に紐づく組織のみアクセス可能
- サービス利用者は組織管理機能へのアクセス不可

### 1.5 非機能要件（non-functional-requirements.md より）

#### パフォーマンス要件
- 初期ページ読み込み時間: 3秒以内
- 画面遷移時間: 1秒以内
- API応答時間: 500ms以内

#### セキュリティ要件
- **認証・認可:**
  - Databricks User認証（Databricks標準機能）
  - ロールベースアクセス制御（RBAC）
  - データスコープ制限（販社ユーザは自社のみ）
- **データ保護:**
  - HTTPS通信
  - データベース暗号化 ←暗号化する？
- **入力検証:**
  - サーバーサイドバリデーション必須
  - SQLインジェクション対策（パラメータ化クエリ）
  - XSS対策（Jinja2自動エスケープ）
  - CSRF対策（Flask-WTF）

#### データスコープ制限要件
- 販社ユーザ: `WHERE sales_company_id = :user_sales_company_id`
- サービス利用者: アクセス不可（グローバルメニューで非表示）

### 1.6 技術要件（technical-requirements.md より）

#### Databricks API認証戦略
- **Databricks Groups API呼び出し:** サービスプリンシパルの認証情報を使用
  - 対象: 組織管理のCUD操作時のグループ作成/更新/削除
  - 理由: Databricksワークスペースのグループ制御には管理者ロールが必須
- **OLTP DBへのマスタCRUD操作:** 共通のDBアカウントを使用
  - 対象: 組織マスタデータ操作
  - データスコープ制限はアプリケーション層で実装
- **Unity Catalogアクセス:** ログインユーザーの認証情報を使用（本機能では未使用）

#### Flask Blueprint構成
```
app/
├── admin/
│   └── organizations.py  # /admin/organizations/* - 組織管理
```

#### データベース
- **使用DB:** OLTP DB (MySQL互換)
- **テーブル:** organizations
- **操作:** SELECT, INSERT, UPDATE (論理削除)
- **トランザクション:** 組織登録・更新・削除時に必要

## 3. 新設した章の詳細

### 3.1 Databricks ワークスペースグループ連携セクション

**新設理由:**
- 組織管理機能の特徴的な要件であるDatabricks連携について、詳細なワークフローとエラーハンドリングが必要
- 標準的なCRUD操作に加えて外部API連携が含まれるため、シーケンス図とロールバック処理の説明が必要

**記載内容:**
- グループ作成API呼び出しのタイミング
- グループ更新API呼び出しのタイミング
- グループ削除API呼び出しのタイミング
- エラー時のロールバック処理
- シーケンス図（組織登録フロー: OLTP DB INSERT → Databricks API呼び出し）

**配置場所:**
- `workflow-specification.md` 内に専用セクションを設ける
- 各ワークフロー（登録/更新/削除）内にDatabricks連携処理を明記

### 3.2 トランザクション管理セクション

**新設理由:**
- 組織登録/更新/削除は、OLTP DBへの操作とDatabricks APIへの操作を含むため、複雑なトランザクション管理が必要
- どのタイミングでコミット/ロールバックするかを明確にする必要がある

**記載内容:**
- トランザクション開始タイミング
- トランザクション終了（コミット）条件
- トランザクション終了（ロールバック）条件
- ロールバック実装例（OLTP DBとDatabricks APIの両方）

**配置場所:**
- `workflow-specification.md` の「トランザクション管理」セクション

### 3.3 データスコープ制限実装詳細セクション

**新設理由:**
- 販社ユーザは自社に紐づく組織のみアクセス可能という要件があるため、WHERE句でのフィルタリング実装が必要

**配置場所:**

## 4. 設計方針

### 4.1 標準的なCRUD画面テンプレートに準拠

組織管理機能は、デバイス管理・ユーザー管理と同様の標準的なCRUD画面構成とします。

**適用するテンプレート:**
- 一覧画面: 検索フォーム + データテーブル + ページネーション
- 登録画面: モーダル形式
- 更新画面: モーダル形式
- 参照画面: モーダル形式
- 削除機能: チェックボックス選択 + 確認モーダル

### 4.2 Databricks連携の明確化

組織登録/更新/削除時のDatabricksワークスペースグループ連携について、以下を明確にします:

**連携タイミング:**
1. 組織登録: OLTP DB INSERT → Databricks グループ作成
2. 組織更新: OLTP DB UPDATE → Databricks グループ更新
3. 組織削除: Databricks グループ削除 → OLTP DB 論理削除

**エラーハンドリング方針:**
- 組織登録時にDatabricks API失敗: OLTP DBをロールバック
- 組織削除時にDatabricks API失敗: 警告表示、OLTP DBはロールバックしない（手動対応）

### 4.3 セキュリティ設計

**データスコープ制限:**
- システム保守者・管理者: すべての組織にアクセス可能
- 販社ユーザ: 自社に紐づく組織のみアクセス可能（sales_company_id でフィルタリング）
- サービス利用者: アクセス不可

**ロールベースアクセス制御:**
- Flask Blueprintレベルでデコレーター実装（`@role_required`）
- 各操作（C/R/U/D）ごとに権限チェック
