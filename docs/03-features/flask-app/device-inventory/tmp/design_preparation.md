## 把握した要件

### 1. 機能概要

**機能ID:** FR-004-5
**機能名:** デバイス台帳管理
**概要:** 物理的なデバイス在庫を管理する機能

### 2. 対象画面

| 画面ID | 画面名 | スラッグ | 概要 |
|--------|--------|---------|------|
| ADM-013 | デバイス台帳一覧画面 | `/admin/device-inventory` | デバイス台帳の一覧・検索/削除 |
| ADM-014 | デバイス台帳登録画面 | `/admin/device-inventory/register` | デバイス台帳の登録 |
| ADM-015 | デバイス台帳更新画面 | `/admin/device-inventory/update` | デバイス台帳の更新 |
| ADM-016 | デバイス台帳参照画面 | `/admin/device-inventory/detail/:id` | デバイス台帳の詳細情報表示 |

### 3. アクセス権限

| 機能 | システム保守者 | 管理者 | 販社ユーザ | サービス利用者 |
|------|--------------|--------|----------|--------------|
| デバイス台帳一覧 | ○ | - | - | - |
| デバイス台帳登録 | ○ | - | - | - |
| デバイス台帳更新 | ○ | - | - | - |
| デバイス台帳削除 | ○ | - | - | - |

**重要:** システム保守者（NSW内部）のみがアクセス可能な機能

### 4. 表示項目（機能要件より）

| No | 項目名 | 説明 |
|----|--------|------|
| 1 | デバイスID | デバイスの一意識別子 |
| 2 | デバイス名 | デバイスの名称 |
| 3 | デバイス種別 | デバイスの種類 |
| 4 | モデル情報 | デバイスのモデル番号等 |
| 5 | SIMID | SIMカードのID |
| 6 | MACアドレス | ネットワークインターフェースの物理アドレス |
| 7 | 在庫状況 | 在庫ステータス |
| 8 | 購入日 | デバイス購入日 |
| 9 | 出荷予定日 | 出荷予定の日付 |
| 10 | 出荷日 | 実際に出荷した日付 |
| 11 | メーカー保証終了日 | メーカー保証の終了日 |
| 12 | ベンダー保証終了日 | ベンダー保証の終了日 |
| 13 | 在庫場所 | デバイスの保管場所 |

### 5. 画面共通要件（CRUD画面テンプレートより）

- 画面サイズは1920×1080
- 画面左部にグローバルメニューカラムを表示
- 登録系のボタンは青系、削除ボタンは赤系
- 登録、更新、参照画面は同画面内でモーダルとして表示
- 論理削除方式
- 削除ボタンは一覧でチェックボックスが選択されたときのみ活性
- 確認モーダルを登録、更新、削除処理実行時に表示

### 6. 非機能要件（実装に関連するもの）

**パフォーマンス:**
- 初期ページ読み込み時間: 3秒以内
- 画面遷移時間: 1秒以内
- API応答時間: 500ms以内

**セキュリティ:**
- ロールベースアクセス制御（RBAC）
- データスコープ制限（organization_closureテーブル使用）
- SQLインジェクション、XSS対策必須

**ユーザビリティ:**
- 一貫したUIパターン
- 明確なフィードバック（成功・エラーメッセージ）
- ローディング表示

### 7. 技術要件（実装に関連するもの）

**フレームワーク:**
- Flask 3.0+ / Jinja2
- Python 3.11
- Flask-WTF（CSRF対策、フォームバリデーション）

**データベース:**
- OLTP DB（MySQL互換）使用
- SQLAlchemyによるORM
- 接続プーリング必須

**認証:**
- Databricksリバースプロキシヘッダ認証
- X-Databricks-User-Id、X-Databricks-Access-Token

## 解析した不足情報

### 1. 検索条件項目が未定義

**現状:** 機能要件書に「検索条件項目: TODO」と記載されている

**設計判断:** 以下の検索条件を設定
- キーワード検索（デバイスID、デバイス名）
- デバイス種別（ドロップダウン）
- 在庫状況（ドロップダウン）
- 在庫場所（ドロップダウン）
- 購入日範囲（日付ピッカー）

**理由:** デバイス台帳管理の一般的なユースケースを考慮し、在庫管理に必要な検索条件を設定

### 2. 在庫状況の選択肢が未定義

**現状:** 「在庫状況」の具体的な値が定義されていない

**設計判断:** 以下の選択肢を設定
- 在庫中
- 出荷済み
- 修理中
- 廃棄予定
- 廃棄済み

**理由:** デバイスのライフサイクルを網羅する一般的なステータス分類

### 3. デバイス種別の選択肢が未定義

**現状:** 「デバイス種別」の具体的な値が定義されていない

**設計判断:** 以下の選択肢を設定
- センサー
- ゲートウェイ
- コントローラー
- その他

**理由:** IoTシステムの一般的なデバイス分類

### 4. バリデーションルールが未定義

**現状:** 各入力項目のバリデーションルールが明示されていない

**設計判断:** UI仕様書にて以下を定義
- 必須項目: デバイスID、デバイス名、デバイス種別、在庫状況
- 最大文字数: 各項目に適切な制限を設定
- 形式チェック: MACアドレス形式、日付形式等

### 5. CSVエクスポート対応

**現状:** FR-004-5にCSVエクスポートの記載なし、ただしFR-004-8で「デバイス台帳」がインポート対象に含まれている

**設計判断:**
- 一覧画面にCSVエクスポートボタンを配置
- 絞り込み条件が適用された状態のデータをエクスポート

**理由:** 他のマスタ管理機能との一貫性を保つため

### 6. ページネーション設定

**現状:** 画面共通要件でページネーションの形式は定義されているが、件数が未定義

**設計判断:**
- 1ページあたりの表示件数: 25件（固定）
- 最大表示件数: 1000件

**理由:** 他の一覧画面との統一性を保つため

## 新設した章の詳細

### UI仕様書での新設セクション

#### 1. CSVエクスポートボタン

**新設理由:** FR-004-8（CSVインポート/エクスポート）の要件に基づき、一覧画面にエクスポート機能を追加する必要があるため

**内容:**
- アクションバーに「CSVエクスポート」ボタンを配置
- 絞り込み条件適用後のデータをエクスポート
- ファイル名形式: `device_inventory_{YYYYMMDD_HHmmss}.csv`

#### 2. 保証期限アラート表示

**新設理由:** デバイス台帳管理の主要目的の一つである保証期限管理を視覚的にサポートするため

**内容:**
- メーカー保証終了日、ベンダー保証終了日が30日以内の場合、警告アイコンを表示
- 期限切れの場合は赤色で強調表示

### ワークフロー仕様書での新設セクション

#### 1. デバイスマスタとの連携確認

**新設理由:** デバイス台帳とデバイスマスタ（FR-004-1）との整合性を確保するため

**内容:**
- 登録時: デバイスIDの重複チェック（デバイスマスタとの整合性確認）
- 更新時: デバイスマスタへの影響確認
- 削除時: デバイスマスタとの紐付け確認

#### 2. 在庫状況変更時の履歴記録

**新設理由:** 在庫管理における監査証跡の確保のため

**内容:**
- 在庫状況が変更された場合、変更履歴をログに記録
- 変更日時、変更者、変更前後の値を記録

## 補足事項

### デバイス管理（FR-004-1）との違い

| 観点 | デバイス管理 (FR-004-1) | デバイス台帳管理 (FR-004-5) |
|------|------------------------|---------------------------|
| 目的 | 運用中デバイスの管理 | 物理的な在庫管理 |
| アクセス権限 | 全ロール（参照のみサービス利用者可） | システム保守者のみ |
| 管理項目 | 設置場所、所属組織 | 在庫状況、購入日、保証期限等 |
| 外部連携 | なし（Azure IoT Hubsへの連携なし） | なし |

### 設計時の留意点

1. **UIの一貫性**: 他のマスタ管理画面（デバイス管理、ユーザー管理等）と同じUIパターンを使用
2. **権限制御**: グローバルメニューでの表示/非表示制御、ルートレベルでの権限チェック
3. **データ整合性**: デバイスマスタとの整合性を考慮した設計
4. **監査対応**: 変更履歴の記録、ログ出力ルールの遵守
