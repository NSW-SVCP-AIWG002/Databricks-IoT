## 1. workflow-specification.md の指摘

### 1-1. mermaid 初期表示 - `SaveCookie` ノードラベルが `after_this_request` パターンと乖離（前回指摘 1-2: 未対応） 🟡

v2.6 で `list_device_inventory` を `after_this_request` パターンに変更した結果、Cookie 操作はレスポンス生成「後」に実行される。しかし mermaid の `SaveCookie` ノードのラベルが「レンダリング直前」のままである。

```
# mermaid 現状（初期表示フロー）
SaveCookie[レンダリング直前<br>Cookieに検索条件を格納<br>set_search_conditions_cookie]
SaveCookie --> Template[Jinja2テンプレートレンダリング...]
```

```python
# v2.6 実装: after_this_request コールバックはレンダリング「後」に実行
@after_this_request
def reset_search_cookie(response):
    """Cookie クリア → デフォルト値で再格納（after_this_request でレスポンス確定後に実行）"""
    response = clear_search_conditions_cookie(response, 'device_inventory')
    response = set_search_conditions_cookie(response, 'device_inventory', search_params)
    return response
```

また、実装では `ClearCookie` と `SaveCookie` が同一の `after_this_request` コールバック内に統合されているが、mermaid では2つの独立したノード（`ClearCookie` はクエリ前、`SaveCookie` はクエリ後）として描画されており、実装構造と乖離している。

**修正案:**
- `SaveCookie` ノードラベル「レンダリング直前」→「レスポンス生成後（after_this_request）」に変更
- `ClearCookie`・`SaveCookie` の2ノードを1つの `ResetCookie` ノードに統合し、`after_this_request` で登録・実行される旨を注記する

### 1-3. 検索フォームバリデーション失敗時に `abort(400)` かつ mermaid の「フォーム再表示」と乖離（前回指摘 1-4: 未対応） 🟡

検索処理ビュー（`search_device_inventory`）のバリデーション失敗時の処理が `abort(400)` であるが、mermaid の `ValidError` ノードでは「エラーメッセージ付きでフォーム再表示」と記述されており、コードと mermaid が乖離している。

いずれかに統一すること:
- **コードを mermaid に合わせる**: `abort(400)` を `render_template(...), 422` パターンに変更
- **mermaid をコードに合わせる**: `ValidError` ノードを「400 エラー返却」に変更し、フォーム再表示しない設計として明記
