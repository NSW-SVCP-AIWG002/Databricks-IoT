# Gold Layer LDPパイプライン設計 - 準備資料

## 1. 把握した要件

### 1.1 機能要件（FR-002-2）

**表示用データ変換・保存処理**:
- **Input**: シルバー層（構造化データ）
- **Output**: ゴールド層（表示用データ）
- **処理内容**:
  - シルバー層からのデータ読み取り
  - ダッシュボード表示用データの準備・保存

### 1.2 非機能要件

| 要件ID | 項目 | 要件値 |
|--------|------|--------|
| NFR-PERF-001 | リアルタイム処理 | 受信から1分以内に可視化可能 |
| NFR-PERF-002 | 処理規模 | 10,000デバイス、1分間隔、10GB/日 |
| NFR-SCALE-002 | データ保持期間 | Gold層: 5年 |

### 1.3 技術要件

| 要件ID | 項目 | 内容 |
|--------|------|------|
| TR-DB-001 | データ処理方式 | LDP（Lakeflow宣言型パイプライン） |
| TR-DB-002 | データストア | Unity Catalog（ADLS/Delta Lake） |
| TR-DB-003 | カタログ構成 | iot_catalog.gold スキーマ |

### 1.4 入力データ（Silver層テーブル）

Silver層から以下のテーブルを入力として使用:

| テーブル名 | 用途 |
|-----------|------|
| silver_sensor_data | センサーデータ（構造化済み） |
| silver_alert_abnormal_state | 異常状態管理 |

### 1.5 出力データ（Gold層テーブル）

ダッシュボード表示用に以下のテーブルを作成:

| テーブル名 | 用途 | 集計単位 |
|-----------|------|---------|
| gold_sensor_hourly | 時間別センサー集計 | 1時間 |
| gold_sensor_daily | 日別センサー集計 | 1日 |
| gold_device_latest_status | デバイス最新ステータス | - |
| gold_alert_summary | アラートサマリー | 日別 |

## 2. 解析した不足情報

### 2.1 要件で未定義の項目

| 項目 | 不足内容 | 設計判断 |
|------|---------|---------|
| 集計粒度 | 具体的な集計期間の指定なし | 時間別・日別を標準として設計 |
| 集計メトリクス | 具体的な集計項目の指定なし | avg/min/max/countを標準として設計 |
| 動的ビュー | 具体的なビュー定義なし | 組織階層フィルタ用ビューを設計 |

### 2.2 Silver層仕様からの推定

Silver層設計書から以下を推定:
- センサーデータ構造: device_id, event_timestamp, sensor_type, sensor_value等
- クラスタリングキー: event_date, device_id
- ストリーミング処理: 10秒間隔のマイクロバッチ

## 3. 新設した章の詳細

### 3.1 動的ビュー設計（新設）

**理由**: FR-006-2「データアクセス制御」の要件を満たすため、組織階層に基づくデータフィルタリング用の動的ビューが必要。

**内容**:
- v_gold_sensor_hourly_filtered: 組織フィルタ付き時間別集計ビュー
- v_gold_sensor_daily_filtered: 組織フィルタ付き日別集計ビュー
- v_gold_device_status_filtered: 組織フィルタ付きデバイスステータスビュー

### 3.2 データ品質チェック（新設）

**理由**: Gold層はダッシュボード表示用のため、データ品質の担保が重要。

**内容**:
- 集計結果のNULLチェック
- 集計期間の完全性チェック
- 異常値の除外ロジック

## 4. 設計方針

### 4.1 アーキテクチャ方針

1. **ストリーミングテーブル**: Silver層からの継続的なデータ取り込み
2. **マテリアライズドビュー**: 集計テーブルの効率的な更新
3. **動的ビュー**: 組織階層に基づくアクセス制御

### 4.2 パフォーマンス設計

1. **Liquid Clustering**: `CLUSTER BY (event_date, device_id)`
2. **パーティショニング**: 日付ベースでのデータ整理
3. **インクリメンタル処理**: 差分更新による処理効率化

### 4.3 保持期間設計

- Gold層データ保持期間: 5年
- アーカイブ戦略: 古いデータは圧縮アーカイブへ移行
