## 📋 概要

このディレクトリには、タスク管理アプリケーションのテスト戦略、テスト実装ガイド、機能別テスト仕様など、テストに関する全てのドキュメントを集約しています。

---

## 📂 ディレクトリ構成

```
docs/06-testing/
├── README.md                         # 本ドキュメント（ディレクトリ構成と利用ガイド）
├── test-strategy.md                  # テスト戦略・方針の詳細ドキュメント
├── test-quality-metrics.md           # テストカバレッジ戦略・品質メトリクス・KPI
│
├── guides/                           # テスト実装ガイド
│   ├── unit-test-guide.md           # 単体テスト実装ガイド
│   ├── integration-test-guide.md    # 結合テスト実装ガイド
│   └── e2e-test-guide.md            # E2Eテスト実装ガイド
│
├── unit-test/                        # 単体テスト仕様（機能ごと）
│   ├── user-management-spec.md      # ユーザー管理機能（Repository層 + UseCase層 + UIコンポーネント層）
│   ├── <機能名>-spec.md             # 他の機能（命名規則: 機能名-spec.md）
│   └── ...
│
├── integration-test/                 # 結合テスト仕様（機能ごと）
│   ├── user-management-spec.md      # ユーザー管理機能 API結合テスト
│   ├── <機能名>-spec.md             # 他の機能（命名規則: 機能名-spec.md）
│   └── ...
│
└── e2e-test/                         # E2Eテスト仕様（機能ごと）
    ├── user-management-spec.md      # ユーザー管理機能 E2Eテスト
    ├── <機能名>-spec.md             # 他の機能（命名規則: 機能名-spec.md）
    └── ...
```

### テスト項目書の命名規則

- **ファイル名**: `<機能名>-spec.md`
- **例**: `user-management-spec.md`, `contract-list-spec.md`, `task-management-spec.md`
- **原則**: 1 機能につき 1 ファイル（テストレベルごとに同じ名前のファイルを配置）

## 📄 各ファイル・ディレクトリの用途

### ルートレベルのドキュメント

| ファイル名                  | 用途                                                                     |
| --------------------------- | ------------------------------------------------------------------------ |
| **README.md**               | 本ドキュメント。ディレクトリ構成と各ファイルの用途を説明                 |
| **test-strategy.md**        | テスト戦略・方針、テストピラミッド、テスト階層ごとの戦略、補完的なテスト |
| **test-quality-metrics.md** | テストカバレッジ戦略、カバレッジ目標値の設定根拠、品質メトリクス・KPI    |

### テスト実装ガイド

| ディレクトリ | 用途                                                             | 主なドキュメント                                                 |
| ------------ | ---------------------------------------------------------------- | ---------------------------------------------------------------- |
| **guides/**  | テスト実装ガイド（各テストレベルの実装方法とベストプラクティス） | unit-test-guide.md, integration-test-guide.md, e2e-test-guide.md |

### テスト仕様（テストレベル別）

| ディレクトリ          | 用途                       | 粒度                                           | 例                      | 命名規則           |
| --------------------- | -------------------------- | ---------------------------------------------- | ----------------------- | ------------------ |
| **unit-test/**        | 単体テスト仕様（機能ごと） | Repository 層、UseCase 層、UI コンポーネント層 | user-management-spec.md | `<機能名>-spec.md` |
| **integration-test/** | 結合テスト仕様（機能ごと） | API 結合、機能結合                             | user-management-spec.md | `<機能名>-spec.md` |
| **e2e-test/**         | E2E テスト仕様（機能ごと） | 画面、ユーザーフロー                           | user-management-spec.md | `<機能名>-spec.md` |

**重要**: テスト項目書は機能ごとに作成し、同じ機能名のファイルが unit-test、integration-test、e2e-test の各ディレクトリに配置されます。実際のテストコーディング時は、プロジェクトの方針に従ってテストファイルを分割してください。

## 🚀 クイックスタート

### ⚠️ 重要：テスト実装前の必読ドキュメント

**テストコードを実装する前に、必ず以下のドキュメントを一読してください：**

📖 **[テスト実装ルール（docs/00-rules/test-rule.md）](../00-rules/test-rule.md)**

このドキュメントには以下の重要な情報が含まれています：

- テストコードの実装例（バックエンド・フロントエンド）
- テスト実行コマンド（Bun 使用）
- テストファイルの配置ルールと命名規則
- テスト項目書とコードのマッピング方法
- CI/CD パイプラインとの統合

### テストを実行する

主な実行コマンド：

- 単体テストの実行: `bun test`
- カバレッジ付き実行: `bun test --coverage`
- E2E テストの実行: `bunx playwright test`
- 特定のテストファイルのみ実行: `bun test <file>`

詳細は [テスト実装ルール](../00-rules/test-rule.md#️-テスト実行コマンド一覧) を参照してください。

## 📚 ドキュメントの使い方

### 1. テスト戦略を理解する

まず、以下のドキュメントを読んで、プロジェクト全体のテスト戦略と方針を理解してください。

- **[test-strategy.md](./test-strategy.md)** - テストピラミッド、テストレベル別の方針（単体テスト、結合テスト、E2E テスト）、補完的なテスト
- **[test-quality-metrics.md](./test-quality-metrics.md)** - カバレッジ目標値、品質メトリクス・KPI、測定方法

### 2. テスト種別ごとの実装ガイドを参照する

実装するテストの種別に応じて、`guides/`ディレクトリ内のガイドを参照してください。

- **単体テストを実装する**: [guides/unit-test-guide.md](./guides/unit-test-guide.md)
- **結合テストを実装する**: [guides/integration-test-guide.md](./guides/integration-test-guide.md)
- **E2E テストを実装する**: [guides/e2e-test-guide.md](./guides/e2e-test-guide.md)

### 3. テスト実装状況・進捗を確認する

機能別のテスト仕様、実装状況、カバレッジは、プロジェクト管理ツール（エクセル/ガントチャート）で管理予定です。

- 各機能のテストケース一覧
- テスト実装状況の確認
- カバレッジ目標と実績の確認
- 実装スケジュールの確認

## 📝 ドキュメント更新ガイドライン

### 新しい機能のテスト仕様を追加する場合

1. **[テスト実装ルール](../00-rules/test-rule.md) を確認**し、ファイル配置ルールと命名規則を理解する
2. プロジェクト管理ツール（エクセル/ガントチャート）に機能を追加し、テスト仕様とスケジュールを定義
3. 必要に応じて、以下のディレクトリにテスト仕様ファイルを作成（命名規則: `<機能名>-spec.md`）
   - `unit-test/<機能名>-spec.md` - Repository 層、UseCase 層、UI コンポーネント層のテスト項目
   - `integration-test/<機能名>-spec.md` - API 結合テストの項目
   - `e2e-test/<機能名>-spec.md` - E2E テストの項目
4. テスト実装時は [テスト実装ルール](../00-rules/test-rule.md) のファイル配置ルールに従う
5. テスト実装後、プロジェクト管理ツールのカバレッジ状況を更新

**例**: ユーザー管理機能の場合

- テスト仕様書: `unit-test/user-management-spec.md`, `integration-test/user-management-spec.md`, `e2e-test/user-management-spec.md`
- 実際のテストコード: テスト実装ルールの配置ルールに従って、適切な粒度で分割

### テスト戦略を変更する場合

1. [test-strategy.md](./test-strategy.md) を更新
2. 関連するガイドドキュメントも更新
