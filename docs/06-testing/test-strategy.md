## 📋 概要

このドキュメントは、ソフトウェアプロジェクトにおける包括的なテスト戦略と品質保証方針を定義します。テストピラミッド、テスト階層ごとの戦略、補完的なテストなど、プロジェクト全体のテスト方針を示します。

**カバレッジ目標値や品質メトリクスについては、別ドキュメント [テスト品質メトリクス・カバレッジ戦略](./test-quality-metrics.md) を参照してください。**

---

## 🎯 テスト戦略の目的

### 1. **品質保証**

- システムの正確性と信頼性を担保
- バグの早期発見と修正
- リグレッション（機能退行）の防止

### 2. **テストカバレッジの可視化**

- テスト結果の透明性確保

### 3. **継続的な品質向上**

- CI/CD パイプラインへの統合
- 自動化による効率化
- テスト駆動開発（TDD）の実践

## 🏗️ テストピラミッド

本プロジェクトでは、効率的なテスト戦略として「テストピラミッド」のアプローチを採用しています。

```
           ┌──────────────────┐
           │    E2E Test      │  少数（クリティカルパスのみ）
           │  (エンドツーエンド) │  ユーザー視点での動作確認
           └──────────────────┘
          ┌─────────────────────┐
          │  Integration Test   │  中程度
          │    (統合テスト)       │  コンポーネント間連携
          │   API統合テスト      │  外部システム連携検証
          └─────────────────────┘
       ┌────────────────────────┐
       │    Unit Test (単体テスト)  │  多数（主要テスト）
       │   個別機能の詳細検証      │  迅速なフィードバック
       └────────────────────────┘
```

### テストの配分比率（目標）

- **単体テスト**: 70%
- **結合テスト**: 20%
- **E2E テスト**: 10%

### テストピラミッドの基本原則

- **迅速なフィードバックと費用対効果の最大化**: 下層のテストほど実行コストが低く、フィードバックが早い。特に単体テストの厚みを確保して、早期に不具合を検知する。
- **レイヤーごとの責務分離**: 各テスト階層は異なる責務と観点を持つ。重複を避け、テストケースの目的を明確にする。
- **自動化と継続的実行**: すべてのレイヤーで自動化を前提にし、CI/CD パイプラインで継続的に実行する。
- **ドキュメント化と可視化**: テストカバレッジ、実行状況、失敗理由をチーム全体で共有できるよう可視化する。

## 🧪 テスト階層ごとの戦略

### 1. 単体テスト（基盤層）

| 項目               | 方針                                                                                                                    |
| ------------------ | ----------------------------------------------------------------------------------------------------------------------- |
| **目的**           | ビジネスロジック、ドメインルール、UI コンポーネント単位の挙動を素早く検証する                                           |
| **対象**           | ビジネスロジック、ドメインサービス、データアクセス層、UI コンポーネント、ユーティリティ関数など副作用の少ないモジュール |
| **ツール（例）**   | Jest / Vitest、JUnit、Mockito、React Testing Library（コンポーネントの単体検証）                                        |
| **実行頻度**       | ローカル開発時のウォッチモード、CI のプルリクエスト検証で常時                                                           |
| **カバレッジ目標** | ステートメント 70%以上（最低）、80%以上（推奨）。重要ユースケースの分岐網羅を優先                                       |

**重要な方針**:

- ビジネスルールの変更は必ず単体テストで担保する。
- 副作用のあるコードはモックを用いて境界を明確にする。
- テストデータはファクトリやフィクスチャを共有化し、重複を排除する。

**対象の詳細**:

- **ビジネスロジック層**: ユースケース、ドメインサービス、エンティティのビジネスルール
- **データアクセス層**: データの永続化、検索、更新処理
- **プレゼンテーション層**: リクエストバリデーション、レスポンス変換
- **認証・認可層**: トークン検証、権限チェック
- **UI 層**: コンポーネントのレンダリング、Props、イベントハンドラー
- **ユーティリティ**: 汎用関数、ヘルパーモジュール

**本プロジェクトのカバレッジ測定方法**:

本プロジェクトでは、バックエンド（Java）とフロントエンド（TypeScript/React）で異なるツールを使用します。

#### バックエンド（`app/task-api`）

| 項目                   | 内容                                                                                                                         |
| ---------------------- | ---------------------------------------------------------------------------------------------------------------------------- |
| **テストランナー**     | JUnit 5                                                                                                                      |
| **カバレッジツール**   | JaCoCo (Gradle)                                                                                                              |
| **測定コマンド**       | `./gradlew test jacocoTestReport`                                                                                            |
| **実行ディレクトリ**   | `app/task-api/`                                                                                                              |
| **測定可能な指標**     | Line Coverage、Branch Coverage、Method Coverage、Instruction Coverage                                                        |
| **カバレッジレポート** | HTML レポート: `build/reports/jacoco/test/html/index.html`<br>XML レポート: `build/reports/jacoco/test/jacocoTestReport.xml` |

**実行例**:

```bash
cd app/task-api
./gradlew test jacocoTestReport
```

**出力例（コンソール）**:

```
> Task :test
> Task :jacocoTestReport

BUILD SUCCESSFUL in 15s
```

**レポート確認**:

HTML レポートをブラウザで開く:

```bash
open build/reports/jacoco/test/html/index.html  # macOS
xdg-open build/reports/jacoco/test/html/index.html  # Linux
```

**JaCoCo の測定指標**:

- **Instruction Coverage**: バイトコード命令のカバレッジ（最も詳細）
- **Branch Coverage**: 条件分岐のカバレッジ（目標: 最低 65%、推奨 75%）
- **Line Coverage**: コード行のカバレッジ（目標: 最低 70%、推奨 80%）
- **Method Coverage**: メソッドのカバレッジ（目標: 最低 75%、推奨 85%）

参考: [JaCoCo Documentation](https://www.jacoco.org/jacoco/trunk/doc/)

#### フロントエンド（`app/task-web`）

| 項目                   | 内容                                                    |
| ---------------------- | ------------------------------------------------------- |
| **テストランナー**     | Jest (Bun)                                              |
| **測定コマンド**       | `bun test --coverage`                                   |
| **実行ディレクトリ**   | `app/task-web/`                                         |
| **測定可能な指標**     | Line Coverage、Function Coverage                        |
| **測定不可な指標**     | Branch Coverage、Statement Coverage（Bun は現在未対応） |
| **カバレッジレポート** | ターミナル出力（表形式）                                |

**実行例**:

```bash
cd app/task-web
bun test --coverage
```

**出力例**:

```
 ----------------------------|---------|----------|---------|---------|
 File                        | % Lines | % Funcs  | % Branch| % Stmts |
 ----------------------------|---------|----------|---------|---------|
  All files                  |   78.45 |   82.50  |    N/A  |    N/A  |
   src/components/           |   85.23 |   88.12  |    N/A  |    N/A  |
   src/lib/                  |   80.15 |   84.33  |    N/A  |    N/A  |
 ----------------------------|---------|----------|---------|---------|
```

**⚠️ Bun の制限事項と対応方針**:

- Bun は現在 **Line Coverage** と **Function Coverage** のみサポート
- **Branch Coverage** は測定不可のため、重要な条件分岐については、テストケース設計時に正常系・異常系の両方を明示的に記述し、手動で確認する
- 将来 Bun が Branch Coverage をサポートした場合、目標値（最低 65%、推奨 75%）を適用予定

参考: [Bun Test Coverage Documentation](https://bun.sh/docs/test/coverage)

### 2. 統合テスト（中間層）

| 項目               | 方針                                                                                                   |
| ------------------ | ------------------------------------------------------------------------------------------------------ |
| **目的**           | 複数モジュールが連携した際の振る舞いと副作用を検証し、単体テストでは拾えない問題を発見する             |
| **対象**           | API クライアントと状態管理の連携、複数層にまたがるユースケース、外部システムとの境界を含むデータフロー |
| **ツール（例）**   | Jest / Vitest + supertest、MSW（Mock Service Worker）、Playwright のコンポーネントテスト機能など       |
| **実行頻度**       | プルリクエスト時、メインブランチマージ時                                                               |
| **カバレッジ目標** | 主要ユーザーフロー（認証、データ CRUD、通知など）をすべてカバー。90%以上（最低）、95%以上（推奨）      |

**重要な方針**:

- MSW や InMemory DB を用いて外部依存を制御し、安定性と速度を確保する。
- ドメインイベントや API 契約変更は統合テストで検知できるようにする。
- 統合テストの数は単体テストの 1/5〜1/10 程度を目安にし、クリティカルなシナリオに集中させる。

**対象の詳細**:

- **API エンドポイント全体**: プレゼンテーション層 → ビジネスロジック層 → データアクセス層 → データベースの一連の流れ
- **認証フロー**: ログイン → トークン発行 → 認証済みリクエスト処理
- **データフロー**: クライアント → API → データベース → レスポンスの往復
- **外部システム連携**: 外部 API やサービスとの統合ポイント

**本プロジェクトのカバレッジ測定方法**:

統合テストでは、**API エンドポイントカバレッジ**を測定します。

| 項目                 | 内容                                                                                                   |
| -------------------- | ------------------------------------------------------------------------------------------------------ |
| **測定指標**         | API エンドポイントのテストケース作成率                                                                 |
| **測定方法**         | (テスト実装済みエンドポイント数 / 全エンドポイント数) × 100%                                           |
| **カバレッジ目標**   | 最低 90%、推奨 95%                                                                                     |
| **測定ツール**       | 手動カウント、または API 仕様書との照合                                                                |
| **実行ディレクトリ** | `app/task-api/src/handler/`（API エンドポイント定義）<br>`app/task-api/src/**/*.test.ts`（統合テスト） |

**測定手順**:

1. `app/task-api/src/handler/` 配下の全 API エンドポイントを列挙
2. 各エンドポイントに対応する統合テスト（`*.test.ts`）の存在を確認
3. テスト実装済みエンドポイント数 / 全エンドポイント数 で計算

**対象**:

- 全公開 API エンドポイント（必須: 100%）
- 主要な内部 API エンドポイント（推奨）

### 3. E2E テスト（最上層）

| 項目               | 方針                                                                                                   |
| ------------------ | ------------------------------------------------------------------------------------------------------ |
| **目的**           | 本番に近い環境でのユーザーストーリー全体の成功/失敗パスを検証し、リリースの信頼性を高める              |
| **対象**           | 重要な業務フロー（ログイン〜データ操作完了、通知の受信、権限管理など）                                 |
| **ツール（例）**   | Playwright、Cypress                                                                                    |
| **実行頻度**       | ナイトリービルド、本番デプロイ前、主要リリースブランチのマージ時                                       |
| **カバレッジ目標** | クリティカルパスの成功系＋代表的な失敗系を最低 1 シナリオずつ。クリティカルパス 100%（必達・妥協不可） |

**重要な方針**:

- テストデータはシナリオごとに初期化し、テスト間の依存を排除する。
- ネットワークや外部 API はできる限りスタブ化するが、本番環境と同等に検証すべき箇所はステージング環境を用意する。
- 実行時間を抑えるため、並列実行と差分実行（タグ管理）を活用する。

**対象の詳細**:

- **クリティカルユーザーフロー**:
  - ユーザー登録 → ログイン → データ作成 → データ一覧表示
  - データ編集 → ステータス変更 → 削除
  - 管理機能（検索 → 詳細表示 → 更新）
- **ブラウザ互換性**: 主要ブラウザでの動作確認
- **レスポンシブデザイン**: デスクトップ・モバイルでの表示・操作性

**クリティカルパス判断指標（Critical Path Criteria）**:

クリティカルパスとは、ユーザーが主要目的を達成するために必須となる操作経路を指します。
以下の指標に基づき、各 E2E シナリオを「クリティカル」または「非クリティカル」に分類します。

| 観点                           | 判断指標                                                     | クリティカルと判断する例                       | クリティカルではないと判断する例      |
| ------------------------------ | ------------------------------------------------------------ | ---------------------------------------------- | ------------------------------------- |
| **ビジネス影響度**             | 不具合が発生するとサービス停止・売上損失・業務不能につながる | ログインできない、決済失敗、データ登録できない | 通知メールが遅延、ヘルプリンクが無効  |
| **利用頻度**                   | 日常的に多くのユーザーが利用する                             | データ一覧表示、データ更新                     | 月 1 回程度しか使われない管理メニュー |
| **代替不可性**                 | 他の手段で代替できない操作                                   | ユーザー登録、データ削除                       | 表示フィルタの切替、テーマ変更        |
| **ユーザー体験の中核性**       | サービス価値を直接提供する操作                               | メイン機能（例：データ管理操作）               | 設定画面の文言変更、プロフィール編集  |
| **外部システム連携の影響範囲** | 外部 API やサービス連携があり、障害時の影響が大きい          | 外部認証（SSO）、外部決済連携                  | SNS 共有、分析イベント送信            |
| **リリースリスク**             | 不具合発生時にリリースを止めるべき重要処理                   | データ登録・更新・削除フロー                   | 軽微な UI 変更、文言修正機能          |

**本プロジェクトのカバレッジ測定方法**:

E2E テストでは、**クリティカルパスカバレッジ**を測定します。

| 項目                 | 内容                                                                 |
| -------------------- | -------------------------------------------------------------------- |
| **測定指標**         | クリティカルユーザーフローのテストケース作成率                       |
| **測定方法**         | (テスト実装済みクリティカルフロー数 / 全クリティカルフロー数) × 100% |
| **カバレッジ目標**   | 100%（必達・妥協不可）                                               |
| **測定ツール**       | 手動カウント、またはユーザーストーリーとの照合                       |
| **実行ディレクトリ** | `e2e/`（E2E テストファイル）                                         |
| **テストツール**     | Playwright                                                           |

**測定手順**:

1. 上記のクリティカルパス判断指標（ビジネス影響度、利用頻度、代替不可性等）に基づき、全クリティカルフローを定義
2. `e2e/` 配下の E2E テストファイル（`*.spec.ts`）で、各クリティカルフローに対応するテストの存在を確認
3. テスト実装済みクリティカルフロー数 / 全クリティカルフロー数 で計算

**本プロジェクトのクリティカルフロー例**:

- ユーザー登録 → ログイン → タスク作成 → タスク一覧表示（`e2e/login.spec.ts`, `e2e/tasks-table.spec.ts`）
- タスク編集 → 完了マーク → 削除（`e2e/tasks-table.spec.ts`）
- 管理者契約検索 → 詳細表示（`e2e/admin-contract-list.spec.ts`）

**ブラウザ・デバイスカバレッジ**:

E2E テストでは、クリティカルパスを複数のブラウザとデバイスで検証します。

**ブラウザカバレッジ（目標: 主要 3 ブラウザ）**:

| ブラウザ   | バージョン | 対応デバイス          | テスト方法                                             |
| ---------- | ---------- | --------------------- | ------------------------------------------------------ |
| **Chrome** | 最新版     | デスクトップ/モバイル | E2E テストツール（例：Playwright、Cypress）            |
| **Edge**   | 最新版     | デスクトップ          | E2E テストツール（例：Playwright、Cypress）            |
| **Safari** | 最新版     | デスクトップ/iOS      | E2E テストツール（デバイスエミュレーション機能を使用） |

**デバイスカバレッジ（目標: デスクトップ/モバイル）**:

| デバイス種別       | 画面サイズ    | テスト内容                           | テスト方法                                                     |
| ------------------ | ------------- | ------------------------------------ | -------------------------------------------------------------- |
| **スマートフォン** | 320px - 767px | モバイル最適化レイアウト、タッチ操作 | E2E テストツール（iPhone、Android エミュレーション機能を使用） |
| **デスクトップ**   | 1024px 以上   | フル機能レイアウト、マウス操作       | E2E テストツール（デスクトップビューポート設定）               |

## 🎨 補完的なテスト

テストピラミッドの 3 層（単体テスト、統合テスト、E2E テスト）は**全プロジェクト必須**の基盤テストです。

以下の補完的なテストは**任意**であり、プロジェクトの性質、規模、要件に応じて選択的に採用することで、品質をさらに向上させることができます。

**導入の判断基準**:

- プロジェクトの属性（B2C/B2B、規模、ユーザー数、セキュリティ要件等）
- チームのリソース（時間、予算、専門知識）
- ビジネス上の優先度（ブランドイメージ、法令遵守、パフォーマンス等）

---

### 1. コンポーネント・ビジュアルリグレッションテスト

| 項目             | 内容                                                         |
| ---------------- | ------------------------------------------------------------ |
| **目的**         | UI 破壊を早期検知し、デザインの一貫性を保つ                  |
| **ツール（例）** | Storybook + Chromatic、Percy、BackstopJS                     |
| **実行頻度**     | プルリクエスト時、デザインシステム更新時                     |
| **対象**         | 再利用可能な UI コンポーネント、主要画面のスクリーンショット |

**推奨プロジェクト例**:

✅ **強く推奨（高い効果が期待できる）**:

- デザインシステムを構築・運用しているプロジェクト
- 複数チームで共通の UI コンポーネントライブラリを使用
- B2C サービスで顧客向け UI の品質・ブランドイメージが重要
- 大規模な Web アプリケーション（複数画面、頻繁な UI 変更）
- コンポーネント駆動開発（Component-Driven Development）を採用

⚠️ **あれば良い（効果は限定的）**:

- 中小規模の Web アプリケーション
- 社内ツール（ただし UI の一貫性を重視する場合）

❌ **不要（投資対効果が低い）**:

- UI がほとんどない API 専用サービス
- プロトタイプや短期プロジェクト
- 小規模な社内ツール（少数ユーザー）

---

### 2. 契約テスト（Contract Testing）

| 項目             | 内容                                                          |
| ---------------- | ------------------------------------------------------------- |
| **目的**         | API クライアントとバックエンド間の契約破壊を防止する          |
| **ツール（例）** | Pact、Spring Cloud Contract                                   |
| **実行頻度**     | API 仕様変更時、プルリクエスト時                              |
| **対象**         | クライアント・サーバー間の API 契約、マイクロサービス間の通信 |

**推奨プロジェクト例**:

✅ **強く推奨（高い効果が期待できる）**:

- マイクロサービス アーキテクチャを採用
- フロントエンドとバックエンドが別チームで独立して開発
- 複数のクライアント（Web、Mobile、Partner API 等）が同じバックエンド API を使用
- API 変更が頻繁に発生するプロジェクト
- チーム間のコミュニケーションコストが高い（地理的分散、タイムゾーン違い等）

⚠️ **あれば良い（効果は限定的）**:

- モノリシックアプリでフロント・バック分離開発
- 中規模の RESTful API プロジェクト

❌ **不要（投資対効果が低い）**:

- モノリシックアプリで単一チーム開発
- API の変更頻度が非常に低い
- プロトタイプや短期プロジェクト
- 小規模チーム（1-3 人）で統合テストが十分に機能している

---

### 3. 非機能テスト

非機能テストは複数の種類があり、それぞれ推奨されるプロジェクトが異なります。

#### 3.1 パフォーマンステスト（負荷テスト、ストレステスト）

| 項目             | 内容                                                                                 |
| ---------------- | ------------------------------------------------------------------------------------ |
| **目的**         | アプリケーションの速度、応答性、スケーラビリティを検証する                           |
| **ツール（例）** | JMeter、k6、Gatling、Lighthouse                                                      |
| **実行頻度**     | スプリント単位、リリース前、本番環境のピーク時想定                                   |
| **対象**         | API エンドポイント、ページ読み込み速度、同時接続数、データベースクエリパフォーマンス |

**推奨プロジェクト例**:

✅ **強く推奨（高い効果が期待できる）**:

- EC サイト（ピーク時の負荷対策が必須）
- 大規模 Web サービス（数十万〜数百万ユーザー）
- 金融システム（レスポンス速度が重要）
- SaaS 製品（複数テナントの同時利用）
- リアルタイム性が求められるサービス（チャット、ゲーム、取引システム等）

⚠️ **あれば良い（効果は限定的）**:

- 中規模の Web アプリケーション
- B2B サービス（ユーザー数が予測可能）

❌ **不要（投資対効果が低い）**:

- 社内ツール（少数ユーザー）
- プロトタイプや MVP
- ユーザー数が非常に少ないサービス（数十人以下）

---

#### 3.2 アクセシビリティテスト

| 項目             | 内容                                                                              |
| ---------------- | --------------------------------------------------------------------------------- |
| **目的**         | 障害のあるユーザーでもアプリケーションを利用できることを検証する（WCAG 2.1 準拠） |
| **ツール（例）** | axe-core、Lighthouse、Pa11y、NVDA/JAWS（スクリーンリーダー）                      |
| **実行頻度**     | プルリクエスト時（自動テスト）、リリース前（手動テスト）                          |
| **対象**         | キーボード操作、スクリーンリーダー対応、色覚異常対応、ARIA 属性、コントラスト比   |

**アクセシビリティ基準**:

- **WCAG 2.1 AA 準拠**: 全ての主要ページで準拠
- **キーボード操作**: 全機能をキーボードのみで操作可能
- **スクリーンリーダー**: NVDA、JAWS での動作確認

**テスト方法**:

| ツール         | 目的                               | 実行頻度     |
| -------------- | ---------------------------------- | ------------ |
| **axe-core**   | 自動アクセシビリティチェック       | E2E テスト時 |
| **Lighthouse** | アクセシビリティスコア測定         | CI/CD        |
| **手動確認**   | スクリーンリーダー、キーボード操作 | リリース前   |

**推奨プロジェクト例**:

✅ **強く推奨（高い効果が期待できる）**:

- 公共サービス・行政サイト（法律で義務化されている国・地域もある）
- 大企業のコーポレートサイト（CSR、ブランドイメージ）
- 教育サービス（学生・教員の多様性）
- 医療・健康サービス（高齢者、障害者ユーザーが多い）
- B2C サービス全般（ユーザーベースの拡大）

⚠️ **推奨（基本的な自動テストは実施すべき）**:

- **ほぼ全ての Web アプリケーション**（axe-core による自動テストは低コストで導入可能）
- 社内ツール（従業員の多様性に配慮）

❌ **不要（投資対効果が低い）**:

- プロトタイプや MVP（ただし製品化時には必須）
- 完全に非公開の開発ツール

---

#### 3.3 セキュリティテスト

| 項目             | 内容                                                                                           |
| ---------------- | ---------------------------------------------------------------------------------------------- |
| **目的**         | 脆弱性を検出し、セキュリティリスクを低減する                                                   |
| **ツール（例）** | OWASP ZAP、Snyk、Dependabot、SonarQube、Trivy                                                  |
| **実行頻度**     | 依存関係スキャン: 毎ビルド、脆弱性スキャン: スプリント単位、ペネトレーションテスト: リリース前 |
| **対象**         | OWASP Top 10 脆弱性、依存関係の既知脆弱性、認証・認可の妥当性、SQL インジェクション、XSS 等    |

**セキュリティテスト項目**:

| カテゴリ               | テスト項目                         | 実施方法   |
| ---------------------- | ---------------------------------- | ---------- |
| **認証・認可**         | JWT 検証、パスワードハッシュ       | 単体テスト |
| **入力バリデーション** | SQL インジェクション対策、XSS 対策 | 結合テスト |
| **CORS**               | オリジン検証                       | 結合テスト |
| **レート制限**         | API 制限の動作確認                 | E2E テスト |

**セキュリティスキャン**:

- **依存関係スキャン**: パッケージマネージャーの監査機能を使用（CI/CD で自動実行）
- **静的解析**: リンター、型チェッカーを使用

**推奨プロジェクト例**:

✅ **絶対必須（全プロジェクトで実施すべき）**:

- **全ての Web アプリケーション**（基本レベルのセキュリティテストは必須）
- 特に以下は高度なセキュリティテストも必須：
  - 個人情報・機密情報を扱うサービス
  - 金融システム（銀行、決済、投資等）
  - 医療システム（個人健康情報）
  - EC サイト（クレジットカード情報）
  - 認証基盤（SSO、OAuth プロバイダー等）

⚠️ **高度なセキュリティテストを推奨**:

- B2C サービス（ユーザー情報漏洩のリスク）
- B2B サービス（企業情報漏洩のリスク）
- 公開 API（外部からのアクセスがある）

❌ **不要なケースはほぼない**:

- セキュリティテストは全ての Web アプリケーションで最低限の実施が必要

---

### 補完的なテストの優先順位（一般的な推奨）

本プロジェクトの性質に応じて、以下の優先順位で導入を検討してください。

| 優先度   | テスト種類                     | 対象プロジェクト         | 導入コスト | 効果       |
| -------- | ------------------------------ | ------------------------ | ---------- | ---------- |
| **必須** | セキュリティテスト（基本）     | 全プロジェクト           | 低         | 極めて高い |
| **高**   | アクセシビリティテスト（自動） | Web アプリ全般           | 低         | 高い       |
| **中**   | セキュリティテスト（高度）     | 個人情報、金融、医療     | 中         | 極めて高い |
| **中**   | ビジュアルリグレッション       | デザインシステム採用     | 中         | 中〜高     |
| **低**   | 契約テスト                     | マイクロサービス、大規模 | 高         | 中〜高     |
| **低**   | パフォーマンステスト           | 大規模、EC、金融         | 高         | 中〜高     |

## 🎓 テスト教育・トレーニング

### 開発者向けガイドライン

**実装前の必須事項**:

- **[テスト実装ルール](../00-rules/test-rule.md)** を必ず一読する
- テストコードの記述例、命名規則、ファイル配置ルールを理解する

**開発フロー**:

1. **新規機能実装時**:

   - 単体テストを先に書く（TDD 推奨）
   - 最低でも正常系・異常系を各 1 つ以上
   - カバレッジ 80%を維持

2. **バグ修正時**:

   - バグを再現するテストを先に書く
   - テストが通ることを確認してから修正完了

3. **コードレビュー時**:
   - テストコードもレビュー対象
   - カバレッジが下がっていないか確認

## 🚀 CI/CD への組み込み

### CI/CD 統合の方針

テストは開発プロセスの各段階で自動的に実行され、品質を継続的に保証します。

### 1. PR 段階

- **単体テストの即時実行**

  - Pull Request 作成時に自動で単体テストを実行
  - テストが失敗した場合、PR マージをブロック

- **結合テストの並列実行**
  - 統合テストを並列で実行し、実行時間を短縮
  - 失敗時は自動でフィードバックを PR コメントに追加

### 2. メインブランチ保護

- **カバレッジ計測と閾値チェック**
  - テストカバレッジを自動測定
  - 設定した閾値（最低 70%、推奨 80%）未満の場合、マージを防止
  - カバレッジレポートを PR コメントに自動投稿

### 3. リリース前

- **ステージング環境での E2E テスト**
  - ステージング環境に自動デプロイ
  - E2E テストと契約テストを実行
  - テスト結果をダッシュボード化し、視覚的に確認可能に

### 4. 運用中

- **ナイトリービルド**
  - 重要シナリオを定期的に実施（毎日深夜）
  - 異常時は Slack/Teams に自動通知
  - トレンドデータの収集と分析

### GitHub Actions ワークフロー例

```yaml
# .github/workflows/test.yml
name: Test Pipeline

on:
  pull_request:
    branches: [main, develop]
  push:
    branches: [main, develop]

jobs:
  backend-test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v3

      - name: Set up JDK 17
        uses: actions/setup-java@v3
        with:
          java-version: "17"
          distribution: "temurin"

      - name: Grant execute permission for gradlew
        run: chmod +x gradlew

      - name: Run tests with coverage
        run: ./gradlew test jacocoTestReport

      - name: Upload coverage to Codecov
        uses: codecov/codecov-action@v3
        with:
          files: ./build/reports/jacoco/test/jacocoTestReport.xml
          flags: backend

  frontend-test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v3

      - name: Setup Bun
        uses: oven-sh/setup-bun@v1
        with:
          bun-version: latest

      - name: Install dependencies
        run: bun install

      - name: Run unit tests
        run: bun test --coverage

      - name: Upload coverage to Codecov
        uses: codecov/codecov-action@v3
        with:
          files: ./coverage/lcov.info
          flags: frontend

  e2e-test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v3

      - name: Setup Bun
        uses: oven-sh/setup-bun@v1
        with:
          bun-version: latest

      - name: Install dependencies
        run: bun install

      - name: Install Playwright
        run: bunx playwright install --with-deps

      - name: Run E2E tests
        run: bunx playwright test

      - name: Upload test results
        if: always()
        uses: actions/upload-artifact@v3
        with:
          name: playwright-report
          path: playwright-report/
```
